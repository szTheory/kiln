## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Inline icon/text separation, focus offsets |
| sm | 8px | Button internals, eyebrow-to-heading nudge, compact control gaps |
| md | 16px | Default element spacing inside cards and between copy blocks |
| lg | 24px | Success panel and detail-section padding |
| xl | 32px | Vertical gap between template header, action row, and success state |
| 2xl | 48px | Major section break inside `/templates` detail when future adjacent blocks are added |
| 3xl | 64px | Page-level top/bottom breathing room; do not exceed within this narrow flow |

Exceptions: none. Defaulted to Kiln's existing 4px rhythm because upstream artifacts did not declare a Phase 24 exception.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Start run |
| Empty state heading | Use template first |
| Empty state body | Apply a built-in template to create a promoted spec, then start a queued run on the saved workflow. |
| Error state | Problem: the template could not be applied or the run could not start. Next step: keep the operator on the same surface, state the failed action directly, and tell them the corrective step in one sentence. |
| Destructive confirmation | none: this phase introduces no destructive action |

Locked microcopy for this journey:
- `Use template`
- `Edit in inbox first`
- `Spec promoted`
- `Start run`
- `Applying…`
- `Starting…`
- `Apply the template first`
- `Run could not start — check workflow fields`

Source notes:
- CTA labels and tone align with `ui-brand.md`, project brand contract, and existing `TemplatesLive`.
- Empty-state copy is a Phase 24 default because upstream artifacts required calm, explicit next-step copy but did not specify the exact non-promoted state message.

## Phase 24 Journey Scope

This contract applies only to the existing operator journey:
1. `/templates`
2. `/templates/:template_id`
3. promoted success panel on the template detail page
4. navigation to `/runs/:id`
5. the destination shell proof at `#run-detail`

Out of scope:
- redesign of the catalog grid, run detail layout, onboarding, inbox, or broader browser-test strategy
- new template capabilities, new run semantics, or deeper run-state visualization
- new banners, modals, or marketing-style callouts

## Component Inventory

### Template detail header

- Keep the existing pattern: `kiln-eyebrow` over `kiln-h1`, one-sentence supporting copy below.
- Header remains border-led, not shadow-led: `border-b border-base-300 pb-4`.
- Supporting copy stays muted (`text-base-content/60`) and one sentence long.
- On `/templates/:template_id`, the primary focal point is the template title and adjacent primary action row; after promotion, the success panel becomes the primary visual anchor.

### Primary action row

- Keep `Use template` as the primary action and `Edit in inbox first` as the adjacent secondary action.
- Both controls stay on the same row and remain visible before promotion.
- Primary button uses `btn btn-sm btn-primary`.
- Secondary button remains bordered and neutral: no accent fill, no warning/destructive styling.

### Promoted success panel

- Keep the inline panel directly below the action row. Do not redirect away on successful promotion for this phase.
- Required selector: `#templates-success-panel`.
- Required style contract: bordered rectangle, no drop shadow, same secondary surface family as the parent detail card.
- Required emphasis: accent border only; avoid success-green takeover. This is a "ready for next action" state, not a celebratory banner.
- Panel content order:
  1. heading: `Spec promoted`
  2. supporting sentence naming the promoted spec and next step
  3. start-run form

### Start-run control

- Required selectors:
  - `#templates-start-run-form`
  - `#templates-start-run`
- `Start run` stays inside the success panel, not in the global header or card footer.
- Button remains primary/accent because it is the immediate next action after promotion.
- Loading label changes to `Starting…` while disabled.

### Run detail destination shell

- Terminal proof for this phase is the existing `#run-detail` container.
- Do not add extra Phase 24-only chrome to the run detail page.
- The destination should feel like a clean continuation of the same operator system: same `kiln-eyebrow`, `kiln-h1`, border-led sections, muted metadata.

## Interaction Contract

### Selector contract

- Operator actions and state transitions use explicit DOM ids as the primary regression seam.
- Required ids to preserve:
  - `#template-card-<template_id>`
  - `#template-use-form-<template_id>`
  - `#template-edit-first-form-<template_id>`
  - `#templates-success-panel`
  - `#templates-start-run-form`
  - `#templates-start-run`
  - `#run-detail`
- Do not replace this journey with text-only selectors or copy-derived test seams.

### State sequence

1. Template detail loads with primary and secondary actions visible.
2. Submitting `Use template` disables only that button and changes label to `Applying…`.
3. On success, the operator remains on the same template detail page and the success panel appears inline.
4. `Start run` is not shown before promotion for the active template.
5. Submitting `Start run` disables only that button and changes label to `Starting…`.
6. Navigation completes to `/runs/:id`.
7. The destination proves success by rendering `#run-detail`.

### Error handling

- Errors remain inline to the journey via flash and preserved page context.
- Do not navigate away on apply/start failure.
- Error copy must name the failed action and the next corrective step.
- Preferred order of specificity:
  1. exact operator correction when known: `Apply the template first`
  2. workflow-shape correction when known: `Run could not start — check workflow fields`
  3. generic fallback only when cause is opaque: `Run could not start`

### Focus, disabled, and hover

- Keep existing Kiln focus treatment from global styles; do not introduce custom Phase 24 focus colors.
- Disabled buttons must remain visible and readable, not hidden or replaced with spinners.
- Hover may strengthen border or contrast slightly; no motion larger than the existing subtle button transition language.
