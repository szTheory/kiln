<decisions>
## Implementation Decisions

### Static site stack (`DOCS-07`)

- **D-9991-01 — Primary stack:** **Astro + Starlight** as a **single meta-project** under **`site/`** (locked at ship). **Starlight** owns **`/docs/**`** (sidebar IA, content collections; content under `site/src/content/docs/`). **Plain Astro pages** own **`/`** (landing: why Kiln, 60–90s story, CTA, brand-locked layout per `prompts/kiln-brand-book.md`).
- **D-9991-02 — Search (v1):** **Pagefind** (or Starlight’s documented default on-site search path) — **no mandatory SaaS search** (Algolia DocSearch deferred until traffic or multi-version needs justify it).
- **D-9991-03 — ExDoc relationship:** **ExDoc / HexDocs remain the canonical home for module API reference** when libraries ship on Hex; this **operator + product docs site** is canonical for **onboarding, architecture narrative, workflow/spec authoring, and configuration reference** (`DOCS-02`–`DOCS-05`). Cross-link explicitly (“API details → HexDocs”) rather than duplicating moduledoc in two places.
- **D-9991-04 — Deferred stack choices:** **Multi-version doc dropdowns** (Docusaurus’s strength) are **out of v1** for this site unless roadmap explicitly reprioritizes parallel major releases. Re-evaluate only if Kiln ships **multiple supported major versions** with materially different operator instructions. **MkDocs Material** is the documented escape hatch if Node contributor friction becomes a blocker — migration would be a deliberate phase, not a parallel experiment.

### Information architecture (`DOCS-01` + docs tree)

- **D-9991-10 — Single deployable:** **One domain, one GitHub Pages site, one build artifact.** Path-based split only: **`/`** = marketing landing; **`/docs/...`** = all structured documentation. **No** `docs.` subdomain split and **no** separate marketing repo for v1 — minimizes drift and duplicate nav (Prisma / Tailwind pattern: product + docs on one origin).
- **D-9991-11 — Operator spine:** Landing CTA and first-run story must route into **`/docs/...`** happy path (zero-to-first-run, then depth). No “orphan” marketing pages without a doc target.
- **D-9991-12 — Future Hex alignment:** If Hex packages ship later, **add HexDocs as a mirror for API** — still **not** replacing the operator site for narrative, diagrams, and cross-cutting configuration docs.

### CI, `mix check`, and publish (`DOCS-06`)

- **D-9991-20 — Social contract for Elixir devs:** **`mix check` and `mix precommit` stay Elixir-only** by default — no required Node/pnpm on the laptop for **application** contributors (principle of least surprise for this repo’s primary language).
- **D-9991-21 — Docs CI workflow:** **Path-filtered** GitHub Actions **`.github/workflows/docs.yml`** for **`site/**`**, `site/.htmltest*.yml`, **`typos.toml`**, and the workflow file itself — **do not** run the docs pipeline on unrelated `lib/` changes. **`pnpm@9`** via **`pnpm/action-setup`**, **`actions/setup-node`** with cache on **`site/pnpm-lock.yaml`**.
- **D-9991-22 — Quality gates (strict on `main`):** After static build emits **`dist/`** (or framework output dir): run **`htmltest`** on built HTML (internal links, assets). Run **`lychee`** on markdown and/or built HTML with **cache**, **timeouts**, **concurrency caps**, and **curated excludes** for known-flaky externals; optionally add a **scheduled** broader external check so PRs stay fast. Run **`typos`** on repo prose (and optionally `docs/**` only first) — prefer **typos** over heavy Node spell stacks for minimal local install story; **`cspell`** remains an allowed upgrade if prose dictionaries fall short.
- **D-9991-23 — Optional local parity:** **`DOCS=1 mix docs.verify`** (`Mix.Tasks.Docs.Verify`) mirrors the CI steps in `site/` (pnpm install, `astro build`, `verify:mermaid`, htmltest script, lychee, typos); **raises** if `lychee` / `typos` missing locally — documented in `CONTRIBUTING.md`.
- **D-9991-24 — Deploy:** **`push` to `main`** → build site → **`actions/upload-pages-artifact`** + **`deploy-pages`** (or equivalent gh-pages publish already described in requirements). Use **`concurrency`** on deploy to avoid overlapping publishes.

### Mermaid and architecture diagrams (`DOCS-04`)

- **D-9991-30 — Authoring format:** Architecture and flow diagrams live as **` ```mermaid `** fenced blocks in Markdown **next to** the prose they explain — diff-friendly, reviewable in PRs.
- **D-9991-31 — Rendering strategy (Starlight-aware):** **Shipped:** **`pnpm run verify:mermaid`** (`site/scripts/check-mermaid.mjs`, **`@mermaid-js/mermaid-cli`** + pinned **`mermaid`** in `site/package.json`) runs in **CI after `astro build`** and on **`mix docs.verify`** — invalid fenced **` ```mermaid `** blocks **fail the gate**, not silently client-only. **Do not** remove this step without replacing it with an equivalent fail-closed check.
- **D-9991-32 — Versioning & theme:** **Pin** `mermaid` (and `mmdc` if used) in the site lockfile; upgrades are intentional with a short “diagram smoke” checklist. Validate diagrams in **light and dark** presentation against brand palette (`prompts/kiln-brand-book.md`); avoid color-only state distinctions; keep diagrams **small** (split four-layer model into multiple diagrams rather than one mega-graph).
- **D-9991-33 — “In sync with code” (practical):** **Convention over codegen for v1:** maintain `docs/architecture/` (or equivalent) **index** (diagram name, file, owning bounded context, last-reviewed). PR template / checklist: **supervision tree or boundary change → update diagram or file a follow-up.** Optional later: codegen from source only where cost/benefit is obvious.

### Claude's Discretion

- Choice between **typos vs cspell** if typos proves insufficient for Kiln jargon; tweaks to **lychee** allowlists as externals flake.
- PR preview hosting (Cloudflare/Netlify) — **nice-to-have** after GitHub Pages path is green; not blocking v1 publish.
