## 1. Static site stack (`DOCS-07`)

| Option | Description | Selected |
|--------|-------------|----------|
| Astro + Starlight | Markdown-first docs + Astro pages for bespoke landing; Pagefind-class search; fast static output | ✓ |
| Docusaurus | Strong versioning + React/MDX ecosystem; heavier; Algolia path well-trodden | |
| VitePress | Fast Vite DX; weaker first-class versioning | |
| MkDocs Material | Excellent prose/docs DX; weaker bespoke marketing unless extra work | |

**User's choice:** Lock **Astro + Starlight** (research + synthesis): best balance for **single combined site**, **search without SaaS**, **performance**, and **solo/small-team maintenance**; defer multi-version docs and Algolia.

**Notes:** ExDoc/HexDocs stay canonical for **API** when packages exist; this site owns **operator narrative + guides + config/architecture prose**.

---

## 2. Information architecture (landing + docs)

| Option | Description | Selected |
|--------|-------------|----------|
| Single domain, path split (`/` + `/docs`) | One build, one GitHub Pages origin; minimal drift | ✓ |
| Subdomain split (`www` + `docs.`) | Independent cadence; more ops/SEO/canonical overhead | |
| Separate repos | Worst drift for solo maintainer | |

**User's choice:** **Single deployable** — Astro landing on `/`, Starlight docs under `/docs`.

**Notes:** Aligns with Prisma/Tailwind-style devtools IA; Phoenix/HexDocs split is acknowledged as **library ecosystem** pattern, not the v1 model for Kiln’s **product** story.

---

## 3. CI, `mix check`, and publish (`DOCS-06`)

| Option | Description | Selected |
|--------|-------------|----------|
| Docs gates only in CI (path-filtered) | `mix check` stays Elixir-pure; strict docs workflow on relevant paths | ✓ |
| Fold full Node docs toolchain into default `mix check` | Single command, high contributor friction | |
| No htmltest / only markdown link check | Misses broken internal links in built HTML | |

**User's choice:** **Path-filtered** `docs-ci` (and deploy workflow): **build → htmltest** on artifact; **lychee** with cache/timeouts/excludes; **typos** (primary) with optional upgrade path to cspell; optional **`DOCS=1 mix docs.verify`** for local parity.

**Notes:** Follows Elixir-upstream pattern of **isolating expensive validation**; avoids “third-party URL flakiness breaks every PR” via caching and optional scheduled broader checks.

---

## 4. Mermaid and diagrams (`DOCS-04`)

| Option | Description | Selected |
|--------|-------------|----------|
| Mermaid in MD + build-time or CI validation | Fails closed on bad syntax; pinned versions | ✓ |
| Client-only Mermaid without CI oracle | Risk: green build, broken runtime diagrams | |
| PlantUML primary | Heavier alternate stack; deferred | |

**User's choice:** **Fenced Mermaid in repo** + **pin versions** + ensure **`astro build` or `mmdc`** proves diagrams — not client-only without validation. **Convention** (`docs/architecture/` index + PR checklist) for staying aligned with code; **codegen** deferred.

**Notes:** Docusaurus has the smoothest first-party Mermaid story; with Starlight, explicitly buy down risk via **build/CLI validation** and small diagrams.
