## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Attach existing repo |
| Empty state heading | Choose how you want to start |
| Empty state body | Use a built-in template for the fastest proof path, or attach one existing repo when you want Kiln to work on code that already exists. |
| Error state | Attach setup is not ready on this host yet. Start from a built-in template now, or return here once local repo attach is available. |
| Destructive confirmation | none in Phase 29 |

Additional locked copy:
- Template framing: `Built-in templates` and supporting line `Fastest way to learn Kiln or prove the first run.`
- Attach framing: `Attach existing repo` and supporting line `Real-project path for bounded work on one codebase you already own.`
- Attach scope line: `Supports a local path, an existing clone, or a GitHub URL. Validation and workspace safety checks happen in the next step.`
- Attach non-goal line: `Attach does not replace the demo/template journey and does not imply that repo validation has already happened.`

Voice rules:
- Short, calm, operational sentences.
- Say what attach is for, what it is not for, and what happens next.
- Avoid `import project`, `connect code`, `sync repo`, or any wording that implies cloning, forking, or validation already exists.

## Phase Surface Contract

### `/onboarding`

- Keep `/onboarding` demo-first and scenario-led. The existing primary route remains `Open recommended template`.
- Add one secondary attach branch near the current next-step CTA cluster and scenario detail panel. It must read `Attach existing repo`.
- The attach CTA must be visually secondary to the template CTA:
  `btn btn-primary` stays on the template path; attach uses bordered secondary treatment with equal height, not ghost text.
- Attach copy must clarify that this is for operators who already have a codebase and want real-project work, not for first proof learning.
- Do not add attach as a scenario, a scenario recommendation, or a `?scenario=` variant.

Required ids:
- `onboarding-attach-existing-repo`
- `onboarding-attach-path-note`

### `/templates`

- Keep `hello-kiln` as the top recommendation and the largest hero on the page.
- Add a peer attach module directly adjacent to or immediately below the first-run hero, above the broader catalog grid.
- The attach module must feel first-class but smaller than the `hello-kiln` hero. It is a parallel branch, not a replacement hero.
- The attach module must include:
  - One-sentence use case
  - Supported source types list
  - Honest scope note that validation arrives in the next phase
  - CTA to `/attach`
- Preserve the current template catalog and template detail behavior unchanged.

Required ids:
- `templates-attach-module`
- `templates-attach-existing-repo`
- `templates-start-choice`

### `/attach`

- Add a dedicated route-backed LiveView at `/attach`.
- Page title: `Attach existing repo`.
- This surface is orientation and routing only. It must not attempt deep validation, hydration, git mutation, or template plumbing reuse.
- Layout shape:
  - Intro hero with attach purpose and trust posture
  - Supported sources panel: local path, existing local clone, GitHub URL
  - What attach means panel: one repo, bounded branch-oriented work, conservative brownfield handling
  - What happens next panel: Phase 30 validation and workspace checks
  - Back-links to `/templates` and `/onboarding`
- If the page includes a placeholder form shell, inputs must be presentational only in this phase and clearly labeled as upcoming, not operational.
- The page must remain useful in an explorable disconnected state. No hard block, no dead-end modal.

Required ids:
- `attach-entry-root`
- `attach-entry-hero`
- `attach-supported-sources`
- `attach-next-step`
- `attach-back-to-templates`

### `/`

- The run board remains a monitoring balcony.
- A convenience attach shortcut is allowed only as a minor secondary link in the journey card or empty state.
- It must never outrank `Browse templates` or become the primary first-use instruction on the board.

Required id if implemented:
- `run-board-attach-shortcut`

## Interaction Contract

- Navigation must be route-based with `<.link navigate={...}>` or `push_navigate`. No modal attach launcher.
- Attach links from `/onboarding`, `/templates`, and optional `/` all land on `/attach`.
- Keep scenario context separate from attach context. Do not reuse `template_id`, `return_to`, or scenario params to encode attach state.
- Cards and panels use rectangles first: `rounded-xl` or `rounded-2xl`, 1px borders, no heavy drop shadows.
- Hover behavior is subtle: border reinforcement or low-opacity background lift only.
- Focus behavior must use the existing warm ring token `var(--ring-focus)` and remain visible on all attach CTAs and back-links.
- Loading state is only needed if a LiveView event is introduced. Use existing muted motion and verb-first labels like `Opening attach flow…`; do not invent spinner-heavy decorative motion.

---

## State Contract

Every new attach-facing module must define these states:

- Default: calm explanatory copy, visible CTA, bordered panel.
- Hover: border or background intensifies slightly; no color inversion.
- Focus: visible warm ring and preserved text contrast.
- Empty: page still explains the two start paths and the attach scope.
- Warning: use warning styling only for honest future-scope notes such as unsupported multi-root/fork behavior.
- Error: use error token only for true unavailability, with a next action back to templates or setup.
- Disabled: avoid disabled-only primary actions; prefer explorable states with exact guidance.
