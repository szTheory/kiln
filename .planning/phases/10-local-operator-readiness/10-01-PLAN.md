# Phase 10 Plan 01 — Local operator readiness

## Objective

Close the gap between “clone exists” and “operator can drive Kiln from LiveView with confidence”: documented path, env checklist, and a **dry-run narrative** that matches `compose.yaml` + host `mix phx.server` (see `.planning/research/LOCAL-DX-AUDIT.md`).

## Falsifiable UAT (manual / scripted)

1. **Fresh env simulation** — From a clean shell (no stale `MIX_ENV`), with only `.env` from `.env.sample` filled (`SECRET_KEY_BASE`, DB URL, `PORT`), `docker compose up -d db` healthy, `KILN_DB_ROLE=kiln_owner mix ecto.migrate` (or `mix setup`) succeeds.
2. **Boot** — `mix phx.server` starts without `KILN_SKIP_BOOTCHECKS` (BootChecks green).
3. **Onboarding** — Browser: `http://localhost:4000/onboarding` loads; wizard can complete **or** surfaces typed blockers for missing `ANTHROPIC_API_KEY` / Docker as designed.
4. **Run board** — After onboarding, `/` shows the run board (empty state acceptable).
5. **Optional traces** — If `docker compose up -d otel-collector jaeger` + `OTEL_EXPORTER_OTLP_ENDPOINT` set, Jaeger receives at least one trace on a trivial action (document exact click path).

## Tasks (execution order)

1. Walk README quick start; note every divergence; patch README or `first_run.sh` comments.
2. Document **when** to run `docker compose up -d dtu` (sandbox stages) vs db-only.
3. Add **operator checklist** (bullet list) to `README.md` or `.planning/research/LOCAL-DX-AUDIT.md` § Runbook.
4. Record **one** `mix test` or integration command that proves local DB + app wiring (`test/integration/first_run.sh` if applicable).

- Running a full external dogfood run (Phase **11**).
