<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator params -> attach intake contract | Untrusted request prose becomes durable spec data and must be validated before persistence. |
| attach intake -> attached repo lookup | The request must bind to the server-owned `attached_repos` row, not reposted hidden fields. |
| mutable draft -> immutable revision | Promotion must copy the approved bounded-request contract without dropping or mutating fields. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-32-01 | T | `Kiln.Attach.IntakeRequest` | mitigate | Enforce required `request_kind`, `title`, `change_summary`, and non-empty `acceptance_criteria` in the changeset before any draft insert. |
| T-32-02 | T | `Kiln.Attach.Intake` | mitigate | Resolve `attached_repo_id` through `Kiln.Attach.get_attached_repo/1` and reject missing rows instead of trusting client-supplied repo metadata. |
| T-32-03 | D | intake payload persistence | mitigate | Trim list fields, validate lengths in the contract changeset, and keep persistence behind `Specs.create_draft/1` rather than ad hoc map writes. |
| T-32-04 | R | draft promotion | mitigate | Copy the bounded-request fields into `spec_revisions` inside the existing `promote_draft/2` transaction so promoted history matches the draft that launched the run. |

</threat_model>
