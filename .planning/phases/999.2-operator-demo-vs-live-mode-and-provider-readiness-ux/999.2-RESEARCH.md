# Phase 999.2 — Technical Research

**Question:** What do we need to know to plan **operator demo vs live mode** and **provider readiness chrome** well?

**Sources:** `.planning/ROADMAP.md` §999.2, `999.2-UI-SPEC.md`, `lib/kiln/model_registry.ex`, `lib/kiln_web/live/provider_health_live.ex`, `lib/kiln_web/components/layouts.ex`, `lib/kiln_web/live/factory_summary_hook.ex`, `lib/kiln_web/router.ex`, `CLAUDE.md` (SEC-01).

## RESEARCH COMPLETE

## 1. Current system

- **`ModelRegistry.provider_health_snapshots/0`** returns a list of maps per provider (`:anthropic`, `:openai`, `:google`, `:ollama`) with `key_configured?`, `recent_error_rate`, `last_ok_at`, spend, etc. Keys come from **`Kiln.Secrets.present?/1`** — never materialized in UI (SEC-01).
- **`ProviderHealthLive`** polls every 5s (`handle_info(:tick)`) and renders RAG borders + status labels (`Operational` / `API key missing` / `Degraded`). Copy explicitly says keys are never shown.
- **Operator shell** is **`Layouts.app`**: logo + nav + **`FactoryHeader`** fed by **`KilnWeb.FactorySummaryHook`** (`on_mount` in `live_session :default`, subscribes to `factory:summary`).
- **No canonical “demo vs live” flag** exists in the codebase today; roadmap/UI-SPEC require a **single source of truth** the shell can read without each LiveView guessing.

## 2. Recommended architecture

| Concern | Recommendation |
|--------|------------------|
| Mode source of truth | New **`Kiln.OperatorRuntime`** (or equivalent) exposing **`mode/0`** → `:demo \| :live \| :unknown`. Load from **`Application.get_env(:kiln, :operator_runtime_mode)`** set in **`config/runtime.exs`** from **`KILN_OPERATOR_RUNTIME_MODE`** (`"demo"` / `"live"` / unset → document default per env). |
| Why not only LiveView state? | Layout and many routes share chrome — **`on_mount` hook** mirrors **`FactorySummaryHook`** so every operator LiveView gets consistent assigns without duplicating logic. |
| Snapshots + polling | Reuse **5s** cadence aligned with **`ProviderHealthLive`** (`@default_poll_ms`). Hook assigns **`operator_snapshots`** from **`ModelRegistry.provider_health_snapshots/0`** on mount + tick. |
| Presentation | New **`KilnWeb.Components.OperatorChrome`** (function components) for mode chip, readiness strip, config presence — keeps **`layouts.ex`** thin; stable DOM ids per UI-SPEC (`operator-mode-chip`, `operator-provider-readiness`, `operator-config-presence`). |
| Copy / a11y | Lock strings and **`aria-label` / `role="status"`** rules to **`999.2-UI-SPEC.md`** — no paraphrase in implementation. |
| Demo semantics | When `mode == :demo`, strip shows **mock/fixture** messaging even if ETS counters show “degraded” — UI-SPEC row “Demo + simulated outage”. |

## 3. Integration points

- **`router.ex`**: add **`{KilnWeb.OperatorChromeHook, :default}`** after **`FactorySummaryHook`** in **`live_session :default`**.
- **Every `Layouts.app` call** in operator LiveViews (and HEEx templates) must pass **`operator_runtime_mode`** and **`operator_snapshots`** from socket assigns (or rely on **attr defaults** only if hook always assigns before first render — **hook assigns on_mount**, so templates should pass **`{@operator_runtime_mode}`** and **`{@operator_snapshots}`** explicitly for clarity; defaults `:unknown` / `[]` avoid crashes if a route omits hook later).
- **Tests:** `Phoenix.LiveViewTest` on **`ProviderHealthLive`** or **`RunBoardLive`** — assert stable ids and absence of secret-like patterns; unit tests for **`OperatorRuntime.mode/0`** parsing.

## 4. Out of scope (defer)

- Toggle UI to switch demo/live at runtime (UI-SPEC: no destructive confirmation — not required v1).
- PubSub push for snapshots (poll is sufficient for parity with `/providers`).
