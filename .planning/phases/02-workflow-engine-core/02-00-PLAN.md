<threat_model>
## Threats
- **T1: Malicious YAML fixture smuggles atom exhaustion vector** (severity: low)
  - Vector: An accidentally-committed `atoms: true` load path on these fixtures could create arbitrary atoms.
  - Mitigation: Task 1 ships only D-55..D-59-shape YAML (no user-controlled field names in fixtures); no code in this plan calls `YamlElixir.read_from_file/2` directly — loading happens in Plan 05 under the canonical `atoms: false` path per D-63.
  - Residual risk: Fixtures are static files under version control, reviewed as part of the normal PR flow.

- **T2: Factory leaks DB rows across tests causing nondeterministic state** (severity: low)
  - Vector: ExMachina `insert/2` without sandbox checkout bleeds state across tests.
  - Mitigation: `Kiln.ObanCase` setup explicitly calls `Ecto.Adapters.SQL.Sandbox.checkout` + `mode({:shared, self()})`; all factories are `ExMachina.Ecto`-based (once Plans 02/03 upgrade the shells) and respect the sandbox owner PID. `Kiln.RehydrationCase` documents its explicit non-checkin pattern.
  - Residual risk: Developer writes a test without `use Kiln.ObanCase` — caught by normal test review.

- **T3: CAS test helper env bleeds into subsequent tests** (severity: low)
  - Vector: `Application.put_env(:kiln, :artifacts, ...)` mid-test that isn't reverted leaves subsequent tests running against an unexpected CAS root.
  - Mitigation: `cas_test_helper.ex` provides a matched `cleanup_tmp_cas/1` helper; consumers MUST call it in `on_exit/1`. The helper captures prior `Application.get_env` value before overriding and restores on cleanup.
