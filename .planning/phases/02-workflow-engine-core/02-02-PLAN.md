    defmodule Kiln.Runs do
      @moduledoc """
      Public API for the runs context. Owns the `runs` table (Plan 02) and, via
      `Kiln.Runs.Transitions` (Plan 06), the state-machine transitions.

      Every public function is a narrow, read-first query + the single insert
      path (`create/1`); state mutation lives in Plan 06.
      """

      import Ecto.Query

      alias Kiln.Repo
      alias Kiln.Runs.Run

      @spec create(map()) :: {:ok, Run.t()} | {:error, Ecto.Changeset.t()}
      def create(attrs) when is_map(attrs) do
        %Run{}
        |> Run.changeset(attrs)
        |> Repo.insert()
      end

      @spec get!(Ecto.UUID.t()) :: Run.t()
      def get!(id), do: Repo.get!(Run, id)

      @spec get(Ecto.UUID.t()) :: Run.t() | nil
      def get(id), do: Repo.get(Run, id)

      @doc """
      Returns every run whose state is NOT terminal — used by `Kiln.Runs.RunDirector`
      to populate its boot-time rehydration list (D-92).
      """
      @spec list_active() :: [Run.t()]
      def list_active do

## Threats
- **T1: SQL injection via state/kind values routed through Ecto.Query interpolation** (severity: low)
  - Vector: A query builder that accepts an arbitrary string for state filtering could be SQLi'd.
  - Mitigation: All state filters use `^active` (a list of atoms validated by Ecto.Enum at changeset boundary) or Ecto.Query pinning. No raw string interpolation into query fragments in this plan.
  - Residual risk: N/A — no dynamic query string assembly.

- **T2: runs.state drift between Ecto.Enum + Postgres CHECK + migration DDL** (severity: medium)
  - Vector: A developer adds `:paused` to Run `@states` without a matching migration → Ecto accepts it, Postgres rejects, transaction aborts at INSERT/UPDATE time, state machine breaks.
  - Mitigation: The migration embeds the state list as DDL CHECK; every schema change requires a matching migration. Test `test/kiln/runs/run_test.exs` asserts `length(states()) == 9`. A drift surfaces immediately in CI as a failed test.
  - Residual risk: Developer can disable the length assertion; review process catches.

- **T3: stage_runs FK on_delete: :restrict creates orphan lock-in on run deletion** (severity: low — by design)
  - Vector: `kiln_app` role + application code may delete a run with live stage_runs; Postgres raises foreign_key_violation which surfaces as Postgrex.Error.
  - Mitigation: This is intentional (D-81 — audit-preserving). `runs` cleanup is a `kiln_owner`-only operation (GC in Phase 5). Per Plan 01-04, kiln_app doesn't have DELETE on `external_operations`; same pattern for runs/stage_runs — grants are INSERT/SELECT/UPDATE only.
  - Residual risk: None — the error is the intended behavior.

- **T4: Workflow checksum forgery via predictable hash input** (severity: low)
  - Vector: `workflow_checksum` is an opaque value. An attacker who can write to `priv/workflows/*.yaml` AND to the `runs` row could forge a matching checksum post-hoc.
  - Mitigation: Value stored at run-start (Plan 08 enqueue path) is `sha256(compiled_graph)` deterministically derived from the YAML on disk at the time. kiln_app has only INSERT/UPDATE on `runs` (not DELETE). Phase 1's three-layer audit enforcement guarantees any tampering attempt is observable in audit_events (run_state_transitioned's payload captures the "before" state).
  - Residual risk: Someone with `kiln_owner` role can bypass this; solo-op trust boundary.
