```elixir
defmodule Mix.Tasks.CheckNoSignatureBlock do
  @moduledoc """
  mix check_no_signature_block — Fails when any workflow YAML under
  priv/workflows/*.yaml has a top-level `signature:` populated with a
  non-null value. D-65 reserves signature for v2 (WFE-02); any v1 workflow
  populating it is a signature-leak bug.

  Emits exit 0 when no offenders; exit 1 + line-listing on violation.
  """

  use Mix.Task

  @shortdoc "Assert no v1 workflow YAML has a non-null signature block."

  @workflows_glob "priv/workflows/*.yaml"

  @impl Mix.Task
  def run(_args) do
    Application.ensure_all_started(:yaml_elixir)

    offenders =
      @workflows_glob
      |> Path.wildcard()
      |> Enum.filter(&signature_populated?/1)

    case offenders do
      [] ->
        Mix.shell().info("check_no_signature_block: OK — no v1 workflow populates signature")
        :ok
      [_ | _] = files ->
        Mix.shell().error("check_no_signature_block: VIOLATION — the following files have a non-null signature block (D-65 reserves signature: null for v1; sign via git commit -S instead):")
        Enum.each(files, fn f -> Mix.shell().error("  - #{f}") end)
        exit({:shutdown, 1})
    end
  end

  defp signature_populated?(path) do
    case YamlElixir.read_from_file(path) do
      {:ok, map} when is_map(map) ->
        # map is string-keyed because yaml_elixir default is atoms: false
        case Map.get(map, "signature") do
          nil -> false
          _ -> true
        end
      _ -> false  # unparseable YAMLs aren't this task's concern
    end
  end
end
```

## Threats
- **T1: Oban pool exhaustion under queue-aggregate 16 + Repo pool 20** (severity: medium)
  - Vector: A workload with all 16 Oban slots active AND LiveView + ops queries spike at the same time → Ecto `DBConnection.ConnectionError` → run stalls with cryptic error.
  - Mitigation: D-68 budget calc is documented in config/runtime.exs comment; real-world usage on a solo-op box rarely saturates 16 slots. Phase 3's `:stages_anthropic`/`:stages_openai` split will raise pool to 28 per D-71. StuckDetector (Phase 5) will detect a run stalled on DB checkout and escalate.
  - Residual risk: A burst of concurrent LV tabs + 10 active runs could momentarily spike — document operational ceiling in README.

- **T2: check_no_signature_block bypass via non-yaml extensions** (severity: low)
  - Vector: Attacker commits `priv/workflows/evil.yml` (single-l) instead of `evil.yaml`; glob pattern `priv/workflows/*.yaml` misses it.
  - Mitigation: The Mix task explicitly globs `*.yaml`. Plan 05's loader also globs `*.yaml` — any `.yml` file is not loaded by Kiln. Document that the canonical extension is `.yaml`.
  - Residual risk: Developer might expect .yml to work — caught by first load attempt failing.

- **T3: Malicious YAML with atom keys crashes mix check_no_signature_block** (severity: low)
  - Vector: The task calls `YamlElixir.read_from_file/2` which defaults to `atoms: false` (string keys) — correct per RESEARCH.md correction #2. If a developer passes `atoms: true` in a copy-pasted version, atom-table exhaustion is possible.
  - Mitigation: Task explicitly uses the default (no options). Acceptance grep `! grep -q "atoms: true" lib/mix/tasks/check_no_signature_block.ex` can be added for defense in depth.
  - Residual risk: Low — atoms: true must be added deliberately.

- **T4: check_bounded_contexts false-positive on module reload during dev iex** (severity: low)
  - Vector: `Code.ensure_loaded?/1` returns false for modules unloaded during `recompile` in iex — not a production concern.
  - Mitigation: Task runs `Mix.Task.run("compile")` first, forcing a fresh compile before the check. Prod CI is not affected.
  - Residual risk: Local-dev iex flake only.
</threat_model>
