# Milestone v0.6.0: Attach existing repo first

**Status:** ✅ SHIPPED 2026-04-24
**Phases:** 29-31
**Total Plans:** 7

## Overview

v0.6.0 turned the first believable local run into the first believable real-project workflow. Instead of expanding into remote control, deploy automation, or broader brownfield shapes, the milestone narrowed on one higher-leverage job for the current solo-operator persona: point Kiln at one existing repository, let it work on a bounded branch, and inspect a conservative draft PR.

## Phases

### Phase 29: Attach entry surfaces

**Goal**: Make attach-to-existing a first-class first-use path instead of an implied advanced workflow hidden behind greenfield-only onboarding.
**Depends on**: Phase 28
**Plans**: 2 plans

Plans:

- [x] `29-01-PLAN.md` — add attach-vs-template branching to onboarding and start surfaces
- [x] `29-02-PLAN.md` — align operator copy and flow guidance around greenfield vs attach entry points

**Details:**
Phase 29 introduced the product-level choice that this milestone is about: operators should see, from onboarding onward, that Kiln can either start from a built-in template or attach to one existing repository. The phase kept the current demo/template path intact while making the attach path honest, explicit, and clearly scoped to real-project work.

### Phase 30: Attach workspace hydration and safety gates

**Goal**: Resolve one attached repository into a safe, usable writable workspace before any coding run mutates git state.
**Depends on**: Phase 29
**Plans**: 3 plans

Plans:

- [x] `30-01-PLAN.md` — accept and validate local-path or GitHub-URL attach sources
- [x] `30-02-PLAN.md` — hydrate or reuse one writable attached workspace with run-scoped metadata
- [x] `30-03-PLAN.md` — refuse dirty, detached, or missing-prerequisite repo states with explicit remediation

**Details:**
Phase 30 handled the brownfield mechanics that the earlier greenfield-first flow never needed. Kiln can now resolve one repository, confirm it is usable, materialize or reuse the right local workspace, and stop early when the repo is unsafe to touch. The phase deliberately excluded multi-root, fork, and clone-to-stack behavior.

### Phase 31: Draft PR trust ramp and attach proof

**Goal**: Carry the attached repo through branch, push, and draft PR orchestration, then close the milestone with explicit automated proof coverage.
**Depends on**: Phase 30
**Plans**: 2 plans

Plans:

- [x] `31-01-PLAN.md` — add run-scoped branch + draft PR orchestration for attached repos
- [x] `31-02-PLAN.md` — add attach happy-path and refusal-case proof coverage and reconcile planning SSOT

**Details:**
Phase 31 turned the attach flow into the intended trust posture: Kiln still works autonomously, but first attached-repo runs land as conservative draft PRs rather than pretending real repositories should behave like built-in templates. The phase also added the owning `mix kiln.attach.prove` command and milestone verification coverage for both success and refusal paths.

## Milestone Summary

**Decimal Phases:**

- None in milestone scope. Backlog phases `999.3` and `999.4` remained outside v0.6.0 execution.

**Key Decisions:**

- `v0.6.0` prioritized attach-to-existing over remote ops, CLI/API work, deploy automation, or backlog cleanup.
- First attached-repo runs default to branch + draft PR without adding synchronous approval gates.
- The first attach milestone is single-repo only; fork, clone-to-stack, multi-root, and reference-repo behavior remain deferred.
- Attach enters through onboarding and template/start surfaces because that is where usefulness is judged first.
- `mix kiln.attach.prove` is the owning proof command for the attach milestone.

**Issues Resolved:**

- Added route-backed attach discovery across `/onboarding`, `/templates`, and `/attach` without displacing the `hello-kiln` first proof path.
- Added canonical attach source resolution, deterministic managed workspace hydration, and typed refusal gates for unsafe repo state.
- Added frozen run-scoped branch delivery and durable push plus draft PR orchestration over the existing GitHub worker seams.
- Closed the milestone with one explicit proof path covering attach happy-path delivery and refusal-case truth surfaces.

**Issues Deferred:**

- `999.4` planning/validation debt cleanup remains backlog work unless a later milestone needs it directly.
- Read-only external reference repos remain a separate follow-on capability.
- Remote operator control plane, deploy/publish flows, fork-and-continue, and deeper brownfield workspace shapes remain deferred beyond `v0.6.0`.
- Pending todo `2026-04-24-review-orphan-phase-03-worktree-residue.md` remains acknowledged deferred work at close.

**Technical Debt Incurred:**

- Repo-wide test execution still emits sandbox ownership noise from spawned role processes even when the suite finishes green.
- The orphan Phase 03 worktree residue todo remains parked rather than being folded into milestone scope.
