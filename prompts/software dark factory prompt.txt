- well tested, specs across happy path, error cases, and boundary cases. focus most energy across happy path and main expected error cases though in terms of integration/e2e style specs which are incredibly important. can mock external deps using our DI with our SRP classes (similar can be done for functions) to make testing easier and faster. all tests especially integration/e2e should be reliable/non-flakey/no-intermittent-failures. using best practices for whatever e2e style testing is used for example playwright best practices.
- CI/CD is a must, use docker/docker compose or whatever to make standing up integration targets and the runtime for test mode easy and reliable/reproducible in both local testing and CI. dogfood the docker setup, great DX (developer experience) and ergonomics

- Domain language: Similarly we always want to maintain an up to date and accurate description ideally in one markdown file of our Domain language, so especially nouns/verbs/events that would happen in the system/use cases/user flows/workflows... this should also serve as source of truth and has the add-on effect of making it very easy for non-technical users to read and understand what is the domain we are working in so that they can become an expert in the domain
- Consider what type of software is being built. Videogame vs. SaaS app vs http server vs embedded software etc. all have very different needs and we should keep that in focus. Obviously forcing DDD into a videogame design doesn't make sense when there are way different access patterns for video games and sometimes OO is even avoided and things like ECS are used... - where applicable I prefer event sourcing or really I mean a ledger of events in other words something that is auditable and can be replayed and used for time series analysis, allows for rewind and easy debugging when you have a "story" of events

gastown/fabro type stuff (we are doing an elixir/phoenix version of dark software factory here):
* every meaningful unit of work should have a durable representation (task / command / bead / workflow node) that can be versioned, resumed, diffed, replayed, and audited. optimize for traceability of work, not just storage of code. Gas Town’s Beads and Fabro’s stage-level git checkpointing both push hard in this direction. 
*docs are part of the runtime of the factory. architecture decisions, domain glossary, BDD/JTBD workflows, failure modes, and operator runbooks should live close to code and be structured so agents can consume them directly. GSD’s current system is explicitly spec-driven and artifact-heavy; your list is already moving in that same direction.
* security defaults should be boring and automatic: CSRF on whenever sessions are used, secure browser headers, scoped data access, audited impersonation, and role-gated debug/admin surfaces. Plug’s CSRF docs still recommend CSRF protection whenever a session is used, including JSON cases.
* use releases and runtime config as the deployment primitive. do not bake environment-dependent secrets or ops config into build-time config. Phoenix’s release docs and current Elixir config docs both push toward runtime configuration via config/runtime.exs.
*prefer append-only history for critical domains. full event sourcing only where replay/audit/time-travel actually pays for the extra complexity; otherwise an append-only audit ledger plus current-state tables is often the better trade. that recommendation follows from the real operational overhead in CQRS/ES tooling: dedicated event stores, handler semantics, projections, and replay/backoff concerns. 
* build a dedicated operator experience. job queues, failed workflows, replays, audits, impersonation traces, rate limits, and “why is this stuck?” should have first-class UI/UX. Phoenix LiveDashboard gives you real-time metrics/debugging and custom pages; Oban Web adds a job dashboard and explicitly recommends securing dashboard access.
* treat observability as a first-class product surface. define a stable internal instrumentation API, emit domain-level telemetry events, propagate correlation ids, and build purpose-specific operator dashboards rather than dumping people into raw logs. Phoenix LiveDashboard supports custom pages, and OpenTelemetry’s Erlang/Elixir docs currently list traces as stable while metrics and logs are still development, which is a good reason to keep your instrumentation boundary vendor- and backend-agnostic.
* prefer explicit domain states over boolean soup. use atoms/enums/state machines for workflow phase, role, status, sync state, review state, etc., so impossible states are harder to represent. Elixir’s design anti-pattern docs explicitly warn about “boolean obsession.”  
* keep the compile graph shallow and intentional. avoid unnecessary macros, watch compile-connected cycles with mix xref, and enforce architectural boundaries in CI. current Elixir docs point to mix xref for spotting compile-connected cycles, and Boundary exists specifically to restrict cross-module dependencies in Phoenix-style apps. 
* on the BEAM, do not organize code around processes unless the runtime properties justify it. processes are for concurrency, ownership, isolation, backpressure, or fault containment — not for pretending every abstraction needs a GenServer. Elixir’s official anti-pattern docs now call this out directly. 
* external integrations should always sit behind behavior-defined ports/adapters. in tests, prefer behavior-based mocks over monkeypatchy/ad hoc mocking. Mox’s current guidance is still: no ad-hoc mocks, only mocks based on behaviours.
* for the dark software factory itself we want phoenix liveview responsive mobile first easy to use
* very important that we are token efficient so use heavier thinking models for planning and faster less thinking models as appropriate for the type of task
* we want smart defaults for the types of models and the graph for how software is built so that it works automatically as much as possible no human intervention is ideal, but can pick from different templates for diff situations/software types if that makes sense
* white-label branding first, very easy to pick colors/typography and logo, but keep it lightweight
* dark mode and light mode out of the box system preferences chosen very readable great UI/UX
* very easy to operate, set it and forget it, easy admin
* every background action should be idempotent, retry-safe, and correlation-id aware. if it sends email, touches billing, mutates external systems, or fans out work, it must tolerate retries and partial failure cleanly
* where applicable... long-running or important side effects should be jobs, not vibes. use durable job execution by default for retries, inspection, and progress reporting; reserve raw tasks/processes for truly local ephemeral work.
* default to server-first UI architecture: LiveView first, function components for reuse, JS only as an escape hatch, streams for large collections, async assigns for slow/remote data. this keeps internal/operator UIs fast and simple without turning the frontend into a second app
* prefer artifact-first execution. plans, design contracts, domain language, acceptance criteria, diagnostics, and postmortems should exist as durable artifacts that agents and humans both read/write. GSD’s current docs explicitly lean on a UI design contract and workflow forensics artifacts
